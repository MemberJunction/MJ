## Why

A new MemberJunction installation is currently delivered as a single zip artifact (`Distributions/MemberJunction_Code_Bootstrap.zip`) that is generated by `CreateMJDistribution.js` and **force-committed into the canonical repo on every release** (~200 commits to date, currently ~31 MB each, and binary so it does not delta-compress). The zip is ~98% migration SQL (93 MB of 95 MB uncompressed); the actual installable code is only ~1.8 MB. The CLI already has the two primitives needed to make the zip obsolete: `mj migrate --tag` clones migrations from GitHub at a ref, and `mj install` bootstraps a working instance from files on disk. This change wires those primitives together so a fresh install pulls exactly what it needs from the canonical repo on demand — eliminating the committed binary, the `CreateMJDistribution.js` pipeline, and the silent-breakage class of bugs that comes with shipping a derived, generated artifact.

## What Changes

- `mj install` becomes a **remote fetcher**. Today it errors unless the code directories already exist on disk (extracted from the zip). It will instead partial+sparse-clone the ~1.8 MB of installable code directories (`MJAPI`, `MJExplorer`, `GeneratedEntities`, `GeneratedActions`, `SQL Scripts` minus `generated/`) from the canonical repo at a target ref, then run the existing bootstrap (npm install/link, CodeGen, env-file generation). **BREAKING**: `mj install` now requires network access to the canonical repo and no longer expects pre-extracted files in the working directory.
- `mj migrate` gains a **baseline-aware partial fetch**. It already clones migrations to a temp dir for `--tag`, but pulls all ~93 MB. It will use a partial clone (`--filter=blob:none`) and check out only the files needed: the most recent Flyway `B`-baseline at or before the target version, plus the `V`-script tail after it. Existing databases fetch only migrations newer than their recorded `flyway_schema_history` version. Fetch is dialect-aware (`migrations/` for mssql, `migrations-pg/` for postgres).
- Both `install` and `migrate` **default to the latest version invariant** = the `main` branch tip (`main` is the published-code branch; `next` is the integration/default branch where new work lands before being merged to `main` for a release). `--tag vX.Y.Z` pins to a specific release for reproducibility. `mj install` records the concrete version it resolved so later `mj migrate` runs stay consistent with the installed code.
- New `mj bundle` command generates a self-contained distribution zip **on demand** (the current `CreateMJDistribution.js` logic, run by whoever needs it) for offline / air-gapped installs. The output is never committed.
- **Removal**: `CreateMJDistribution.js`, the committed `Distributions/MemberJunction_Code_Bootstrap.zip`, the `!MemberJunction_Code_Bootstrap.zip` un-ignore rule, and the release-pipeline step that regenerates and commits the zip.

Out of scope: baseline *production*. The `B`-baseline scripts are already generated and validated elsewhere (`create-new-baseline-migration` command + `baseline-roundtrip` workbench). This change only *consumes* baselines on the fetch side. The canonical repo continues to keep the full migration history as the source of truth.

## Capabilities

### New Capabilities
- `cli-remote-install`: `mj install` resolves a target ref (default latest, `--tag` to pin), partial+sparse-clones the installable code directories from the canonical repo, runs the existing bootstrap, and records the resolved version for later migrate consistency.
- `cli-migration-fetch`: `mj migrate` selects migration files to fetch based on the most recent baseline at/before the target (fresh DB) or the database's current applied version (existing DB), fetching only that slice via partial clone, dialect-aware.
- `cli-distribution-bundle`: `mj bundle` produces a self-contained installer zip on demand from a target ref, replacing the committed artifact for offline/air-gapped use.

## Impact

- **Package**: `packages/MJCLI` — `src/commands/install/index.ts`, `src/commands/migrate/index.ts`, `src/config.ts`, new `src/commands/bundle/index.ts`, and a shared remote-fetch helper.
- **Repo / release pipeline**: removal of `CreateMJDistribution.js`, the committed `Distributions/*.zip`, the `Distributions/.gitignore` un-ignore rule, and the CI step that commits the zip. Stops ongoing ~31 MB/release growth of repo history.
- **Config**: `mj.config.cjs` gains a pinned-version field (resolved ref) and the migrate source default shifts from `filesystem:./migrations` to remote-at-latest.
- **Operational**: `install` and `migrate` now require network access to GitHub by default; offline installs route through `mj bundle`. Migrations are no longer present on disk in a fresh install unless bundled.
- **Branch model (resolved)**: the "latest version invariant" default targets `main` (published code). `next` is the integration branch and is not a default install/migrate target.
