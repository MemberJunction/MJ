UX mockup for the proposed restore feature. Six scenes, top to bottom — open in a browser and scroll.

1

Timeline — restore action on every change type

The slide-out panel today hides the restore button on Create / Delete / Snapshot rows. The proposed UX shows a restore action (relabeled appropriately) on every row, and adds a Restored filter pill alongside the existing type filters.

2

Restored-from lineage chip

When a change row has RestoredFromID populated (i.e., it was created by a restore), a violet chip identifies the source version. Clicking the chip jumps to and highlights the source row in the same timeline.

3

Restore confirm modal — full-record diff with field opt-out

Compares the live record to the chosen snapshot field-by-field. Pre-checked rows = fields that will change. Each row has a checkbox so the user can partially restore (e.g., revert the description but keep the new status). Unchanged fields collapse into a disclosure to keep the diff focused.

4

Un-delete entry point — Recycle Bin chip on entity grid

Hard-deleted records aren't visible in the live grid, so the user needs an entry point. Proposed: a small violet chip in the grid header, visible only when the entity has TrackRecordChanges = 1 AND the user has CanCreate. Soft-deletes don't need this — the standard restore flow on the soft-deleted record handles them.

5

Recycle Bin view

Each card represents the most recent Delete change for a unique RecordID, with key identifying fields parsed from the FullRecordJSON snapshot. [Restore] opens the re-create dialog (next screen). [View history] drops the user into the full timeline for that deleted record.

6

Re-create from snapshot — un-delete dialog

The un-delete path is an insert, so there's no current-vs-snapshot diff — the live record is gone. Instead, we show the snapshot the new record will be created with, plus a PK-collision warning if a record with the same ID already exists.
