Typed JSON Extensions — Architecture

One reusable engine. Lists are the first consumer; any entity's typed JSON column gets the same treatment. new reuse existing

Phase 0 · Runtime engine — @memberjunction/json-schema-forms

Framework-agnostic. Pure TypeScript, fully unit-tested. Shared by server + Angular.
ajv 8 (already a dep) schema parse + validate schema → FormQuestion[] mapper virtual-field enumerator ResolvedSchema abstraction (fixed | polymorphic)

Phase 1 · Reusable Angular components

Thin. Bind a schema + value, render controls, emit changes.
<mj-schema-form> <mj-schema-editor> DynamicFormFieldComponent (14+ controls) FormQuestion interface credential-type-edit-panel pattern

Phase 2 · Data model (additive only)

Migrations + CodeGen. Complies with publish-then-no-break policy.
MJ: List Types (entity) List.ListTypeID List.ExtendedData ListDetail.AdditionalData (reused) EntityField.ValueJSONSchema

Phase 3 · Lists consumer

Polymorphic binding via ListTypeID.
ListEntityServer.ValidateAsync ListDetailEntityServer.ValidateAsync List form: <mj-schema-form> List Detail grid: editable attrs

Phase 4 · View virtual columns + filtering

Fixed binding via EntityField.ValueJSONSchema. Independent ship.
ViewGridColumnSetting.jsonPath grid valueGetter filter virtual fields SQLDialect.JsonExtract (already exists)
Why this order. Phases 0–3 deliver the Lists feature the team asked about. Phase 4 is the general "JSON sub-fields as columns" capability and can ship on its own timeline — it only depends on the Phase 0 engine and the additive EntityField.ValueJSONSchema column.
