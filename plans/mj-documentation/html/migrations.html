Database migrations

MJ uses Flyway for forward-only migrations against SQL Server and PostgreSQL. CodeGen owns several columns and behaviors — knowing what to leave to CodeGen prevents the most common migration pitfalls.

Rendered from migrations/CLAUDE.md

Overview

MJ ships its core schema migrations under migrations/v5/. Your application adds its own migrations to the highest-numbered v* folder. Migrations are forward-only — no down migrations. Roll forward with a corrective migration if needed.

Flyway model

Migrations apply in filename order. Each migration runs in its own transaction (where the database supports DDL transactions). Migration tracking lives in __mj.flyway_schema_history.

# Apply pending migrations
mj migrate up

# Show current status
mj migrate info

# Validate that local files match what's been applied
mj migrate validate

Naming convention

Use this exact format:

VYYYYMMDDHHMM__v[VERSION].x_[DESCRIPTION].sql

# Example
V202605290930__v2.46.x_add_project_status_column.sql

What CodeGen owns — DON'T put these in your migration

CodeGen will conflict with these

Including any of the items below will cause CodeGen failures, duplicate indexes, or migration conflicts during the next mj codegen run.

  • Timestamp columns — __mj_CreatedAt and __mj_UpdatedAt are added by CodeGen with proper defaults and triggers. Don't add them manually.
  • Foreign key indexes — CodeGen creates these as IDX_AUTO_MJ_FKEY_<table>_<column>. Don't create FK indexes manually.
  • Stored procedures & views — CodeGen generates spCreate*, spUpdate*, spDelete*, and views per entity.

SQL Server ↔ PostgreSQL

MJ supports both engines. Migrations use the ${flyway:defaultSchema} placeholder and idiomatic SQL for each engine.

UUIDs are case-different

SQL Server returns UUIDs uppercase; PostgreSQL returns them lowercase. Use UUIDsEqual() instead of === in TypeScript code that runs against both. See the UUID Comparison guide.

Seed data — use mj-sync, not SQL INSERTs

When you create a new lookup or reference table, never seed it with SQL INSERTs in the migration. Create a metadata/ directory for the entity with declarative .json records, then push:

npx mj sync push --dir=metadata --include="agent-request-types"

The metadata-sync system handles upserts cleanly, supports @lookup: references to entity names, and is safe to re-run. SQL INSERTs in migrations cause duplicate-key errors and can't reference other entities by name.

Publish-then-no-breaking-changes policy

Within a published OpenApp major version, only additive schema changes are allowed:

  • ✅ Adding new tables, columns, indexes
  • ✅ Widening types (NVARCHAR(50) → NVARCHAR(200))
  • ✅ Adding optional foreign keys
  • ❌ Dropping tables or columns
  • ❌ Narrowing types
  • ❌ Renaming
  • ❌ Adding required parameters to existing sprocs

Breaking changes force a major version bump. The full policy is in PUBLISH_NO_BREAK_POLICY.md.
