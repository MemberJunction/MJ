Architecture

MemberJunction is a metadata-driven Common Data Platform. The schema you put in the database becomes the application — typed entities, GraphQL APIs, Angular forms, validation, audit, and AI capabilities all derive from it. This page walks through the seven layers that make that possible.

Rendered from MJ/README.md + hand-authored prose

Overview

MemberJunction is split into seven cooperating layers. The lower three (Data & Communication, Metadata, Entity & Actions) form the platform's runtime. The upper three (API & Protocol, AI, Client) expose it to humans, agents, and other systems. The seventh — CodeGen & Tooling — sits across the middle, regenerating the artifacts that connect all the others whenever your schema changes.

The whole system is held together by one idea: the database is the source of truth for application shape. Add a column with the right metadata, and an Angular form field, a GraphQL endpoint, an entity TypeScript property, validation, audit, and AI metadata all show up. You don't write that integration code — CodeGen writes it for you.

The seventh layer wraps the rest

CodeGen & Tooling — CodeGenLib, MetadataSync, SchemaEngine, DBAutoDoc, and migrations (Flyway) — regenerates entity classes, sprocs, views, GraphQL schemas, and Angular forms whenever the database schema changes. It's how the layers above stay aligned with the database without manual integration code.

The metadata layer

Metadata is the system's central nervous system. Every entity, field, relationship, validation rule, permission, dashboard, AI agent, prompt, and action lives in metadata tables. Those tables are described by metadata too — MJ describes itself.

At runtime, the Metadata class loads this information from a provider (SQL Server, PostgreSQL, or GraphQL) and exposes it to the rest of the system. Application code never instantiates an entity directly:

// Always go through the metadata layer — never `new UserEntity()`
const md = new Metadata();
const user = await md.GetEntityObject<UserEntity>('Users');

This indirection is what lets the same code run on the server (going against SQL Server / PostgreSQL directly) and in the browser (going through GraphQL) without changing application logic.

Provider architecture

MJ has three concrete data providers and a clean abstraction (IMetadataProvider) over them. Each provider handles authentication, query translation, mutation, and caching for its target environment.

Provider Where it runs Talks to
SQLServerDataProvider MJAPI & Node scripts SQL Server directly (mssql driver)
PostgreSQLDataProvider MJAPI & Node scripts PostgreSQL directly (pg driver)
GraphQLDataProvider Angular Explorer & browser apps MJAPI's GraphQL endpoint

A single application registers one provider as the "global default" via Metadata.Provider. Multi-tenant or multi-server clients can hold several provider instances simultaneously — see the caching guide for how cross-provider invalidation works.

Runtime data flow

A typical entity read from a browser:

  1. Angular component calls rv.RunView({ EntityName: 'Users', ... })
  2. The GraphQL provider checks its local cache. Hit → returns plain objects, transforms to BaseEntity if requested.
  3. Miss → sends a single GraphQL query to MJAPI.
  4. MJAPI's resolver runs through the SQL Server / PostgreSQL provider, which also checks its server-side cache.
  5. Hit → returns cached JSON. Miss → executes the view stored procedure, caches the result, returns it.
  6. Result travels back up the stack, hydrating into typed UserEntity instances on the client.

Writes are the same pattern in reverse, with one addition: every successful Save() publishes a BaseEntityEvent. Subscribers — including the server's own cache layer and any client that has cached this entity — receive the event and either update in place or invalidate. This is how the system stays coherent across servers and browser tabs.

Protocols

GraphQL

The primary protocol. Apollo Server, auto-generated schemas from entity metadata, strongly-typed clients in TypeScript and Angular.

MCP

Model Context Protocol server. Exposes entities, actions, and queries to Claude Desktop, Cursor, and any MCP-compatible client.

A2A

Agent-to-Agent protocol. MJ's agents can be invoked by, and can invoke, agents on other systems that speak A2A — including other MJ deployments.

The class system

MJ uses a dynamic class factory (MJGlobal.ClassFactory) wired up via the @RegisterClass decorator. Generated entity classes register themselves under their entity name; custom subclasses can register for the same name with higher priority. Whoever registered most recently with the highest priority wins.

This is how, for example, you can ship a UserEntityExtended in your application that overrides Save() on the Users entity without modifying generated code. The metadata layer asks the factory for "the right class for Users" and your subclass comes back.

Class manifest required for tree-shaking

Because ClassFactory uses dynamic registration, modern bundlers (ESBuild, Vite) can't see the classes as used and may tree-shake them. The mj codegen manifest command generates a static CLASS_REGISTRATIONS array that the bundler does see — preventing accidental class loss. See the lazy loading guide.

Caching & pub/sub

MJ's caching is multi-tier and event-driven:

  • Server cache — every RunView checks first. Small unfiltered results auto-cache with in-place upsert/remove on entity events.
  • Filtered/sorted caches are invalidated (not updated) on entity changes — JS can't evaluate SQL predicates safely.
  • Client cache in GraphQLDataProvider follows the same model.
  • Redis pub/sub (optional) propagates entity events across multiple API server instances.
  • BaseEngine integration — engine classes that cache entity arrays auto-subscribe to BaseEntityEvent for their configured entity. Save / delete / remote-invalidate triggers an in-place array update and emits to RxJS observers.

The full system is documented in the Caching & Pub/Sub Guide — about 24 minutes of reading and the single most important architecture document for anyone building on MJ at scale.
