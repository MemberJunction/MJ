Contribute code

Contributing

The MJ codebase has strong conventions. The fastest way to a merged PR is to follow them. The high-level rules:

  • Branches: Cut from next with a descriptive name. Track the same-named remote. Never push directly to next or main.
  • Types: No any. No .Get('FieldName') when a typed property exists. Strong typing throughout — generated entity classes are the source of truth.
  • Tests: Vitest. When you change a package, run its tests. Broken tests are as bad as broken builds.
  • Imports: No dynamic import() as a shortcut. Static imports at the top of files. The narrowly-justified exceptions are listed in CLAUDE.md.
  • Re-exports: Don't re-export types from other packages. Import directly from the source.
  • Singletons: Use BaseSingleton<T>, not static _instance.
  • Preferences: Use UserInfoEngine, not localStorage.
  • Migrations: Read migrations/CLAUDE.md before authoring SQL.

For the full developer handbook, see CLAUDE.md in the repo. It's exhaustive but worth the read if you'll be making non-trivial contributions.

PR workflow

From idea to merge

1

Open a discussion or issue first

Especially for non-trivial changes

Saves you (and reviewers) from rejected PRs. Architectural changes, new packages, or large refactors should always be discussed before coding.

2

Cut a feature branch from next

Track the same-named remote

Use the new-branch Claude Code skill or do it manually: git checkout -b feature/x && git push -u origin feature/x.

3

Add a changeset

One per logical change

Run npx changeset (or use the changeset skill). Required for all PRs that touch published packages.

4

Open a PR against next

Use the PR template

Include a clear test plan. The create-pr Claude Code skill builds a strong starting point.

5

Address review

CI must pass, tests must pass

Push fix commits rather than amending. Squash happens at merge time.
