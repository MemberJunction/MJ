# Returning-Visitor Memory — Cross-Session Continuity for Realtime Sessions & the Widget

**Part of:** [Agent Bridges & Public Widget program](./README.md)
**Goal:** When a visitor talks to an agent (widget text/voice, telephony, any realtime surface) and comes back later, the agent has their prior context — a recap of past conversations, scoped to *who that visitor turns out to be*. Off by default, per-widget opt-in.
**Status:** Design proposal (incorporates GH-thread feedback: polymorphic resolved identity)

---

## 0. The one-paragraph thesis

We are **not** building a new memory engine or a session-graph rewrite. Returning-visitor memory is three small additions on existing rails: **(a)** a durable visitor anchor (a cookie) for the *unresolved* anonymous state, **(b)** a **polymorphic resolved-identity** pointer (`EntityID` + `RecordID`) on the conversation for the *resolved* state, and **(c)** recaps written as **agent memory notes** that the existing session-prep injection already pulls into context. A per-widget toggle gates all of it. The agent runtime, the realtime pathway, and the memory system are unchanged in shape — we add identity plumbing and one summarize-on-close step.

## 1. Current state (verified against code)

### 1a. `MJ: AI Agent Sessions` already self-references — but it's resume-scoped, not cross-visit

```
AIAgentSession fields (verified, entity_subclasses.ts):
  ID, AgentID, ConversationID, UserID, Status,
  LastSessionID        -- self-FK → AIAgentSession.ID
  RootLastSessionID     -- auto root-ID of the LastSessionID chain
  HostInstanceID, LastActiveAt, ClosedAt, CloseReason,
  RecordingFileID, RecordingStartedAt, RecordingMedia,
  OriginStepID, DestinationStepID, Condition
```

`LastSessionID` exists, but its meaning is **session resumption** — it stitches reconnect *legs of one logical session* (janitor restart, dropped socket) back together. Confirmed by the consumers:

- `realtime-session-review.service.ts`: *"legs together (newest leg's `LastSessionID` → prior leg); the review loader walks that … CHAIN-AWARE: when the reviewed session carries a `LastSessionID` (it RESUMED an earlier …)"*
- `realtime-session-data.ts`: `IsResumed: session.LastSessionID != null` / *"Session resumed from a prior one."*

> **Do not overload `LastSessionID` for returning visitors.** It owns reconnect semantics and the replay viewer walks it. A returning visit is a *different* relationship (new logical session, possibly days later) and belongs at a different altitude (see §2).

### 1b. Agent memory already does recaps — scoped to `UserID` today

```
MJ: AI Agent Notes fields (verified):
  AgentID, UserID, CompanyID,
  SourceConversationID, SourceConversationDetailID, SourceAIAgentRunID,
  Note, Status (Provisional→Active→Archived), Type, IsAutoGenerated,
  EmbeddingVector, EmbeddingModelID
```

The memory system (see [AGENT_MEMORY_GUIDE](../../../guides/AGENT_MEMORY_GUIDE.md)) already: auto-generates notes, runs the Provisional→Active→Archived lifecycle, consolidates/decays via the Memory Manager, and **injects relevant notes at session prep**. A "recap of the prior conversation" is exactly an auto-generated memory note. The single limitation: notes scope to a `UserID`, which presumes the counterparty is a `__mj.User`.

### 1c. The widget knows guests only as an ephemeral per-session id

`Conversation.ExternalID` is stamped with an **opaque per-session** guest id for RLS isolation (`types.ts`: *"Opaque per-session id … so the Widget Guest RLS filters isolate this guest"*). Nothing durable survives a page close — there is no concept of "the same visitor, a week later."

### 1d. Per-widget config lives on `MJ: Widget Instances`

```
Widget Instances fields (verified, metadata):
  Name, PublicKey, ApplicationID, PinnedAgentID, GuestRoleID,
  AllowedOrigins, Modality, AuthStrategy, Status,
  SessionTTLMinutes, RateLimitPerMinute, VoiceMaxSessionMinutes
```

The "Remember returning visitors" toggle is a new field here — consistent with how the other widget behaviors are configured.

### 1e. The polymorphic `EntityID` + `RecordID` pattern is standard MJ

A real FK to `__mj.Entity.ID` plus an indexed (non-FK) `RecordID nvarchar(450)` is used across the codebase (Record Changes, etc.). This is the established way to point at "any record of any entity."

## 2. Design decisions (locked for this feature)

| # | Decision | Chosen | Why |
|---|---|---|---|
| R1 | **Resolved identity is polymorphic, not a User FK** | `ResolvedEntityID` (FK → `__mj.Entity`) + `ResolvedRecordID` (`nvarchar(450)`, indexed, no FK) | "The conversation is always with a `__mj.User`" is a false premise. A verified visitor may resolve to `__mj_BizAppsCommon.Person`, a member/contact record, or a `User`. Pointing at a User is one case, not the contract. |
| R2 | **Chain at the conversation altitude, not the session** | `Conversation.PreviousConversationID` (self-FK) + auto root | History and memory are conversation-scoped; a session is an ephemeral connection leg. A session-level pointer duplicates state and collides with `LastSessionID`'s resume semantics. |
| R3 | **Durable visitor anchor = a cookie value, distinct from the session id** | `Conversation.VisitorKey` (`nvarchar`, indexed); the cookie holds the value | The unresolved/anonymous state needs *something* stable to find prior conversations by. `ExternalID` is per-session and stays as-is for RLS. |
| R4 | **Recap = an agent memory note, not a new store** | Summarize-on-close → Active memory note; existing injection pulls it in | Reuse the Memory Manager + injection. No parallel recap table, no new injection path. |
| R5 | **Memory scoping becomes entity-agnostic** | Add `ScopeEntityID` + `ScopeRecordID` to `MJ: AI Agent Notes` (nullable; `UserID` retained) | So a recap can attach to *any* resolved record, matching R1. (Smaller-scope fallback in §6.) |
| R6 | **Off by default, per-widget opt-in, with retention + consent** | `Widget Instances.RememberReturningVisitors` (default 0) + a retention TTL; visitor-facing notice + "forget me" | A tracking cookie plus a stored recap of an anonymous person is a consent surface. |

## 3. Target architecture

```
Visit #1 (anonymous)                         Visit #2 (returning, same cookie)
──────────────────                          ─────────────────────────────────
cookie minted (VisitorKey)                  cookie presented → resolve prior
  │                                            conversations by VisitorKey
  ▼                                            │
Conversation A                                ▼
  ExternalID  = <per-session, RLS>          new Conversation B
  VisitorKey  = <durable cookie>              VisitorKey            = <same cookie>
  Resolved*   = null (anonymous)              PreviousConversationID = A
  │                                           Resolved*             = carried from A (if any)
  │  …conversation happens…                   │
  ▼  on close:                                ▼  session prep:
Memory Manager summarizes A →               memory injection pulls A's recap note
  AI Agent Note (Active, IsAutoGenerated)     (resolved via VisitorKey / Resolved* pair)
  Scope* = Resolved* pair (or UserID)         → agent greets with prior context
```

**Resolution lifecycle:** `anonymous (cookie only)` → *optional* magic-link verify / host-passed identity → `resolved (EntityID, RecordID set)`. On resolution, the visitor's prior conversations (found by `VisitorKey`) re-key onto the resolved pair, and any visitor-scoped memory merges onto it. This is the same "attribute back to a known identity" move from the widget-auth work, now spanning visits — and it attributes back to a **record**, not specifically a user.

### 3a. Schema additions

**`MJ: Conversations`** (the "who/what this thread is with" + the chain):
```sql
ALTER TABLE ${flyway:defaultSchema}.Conversation ADD
    VisitorKey             NVARCHAR(255) NULL,
    ResolvedEntityID       UNIQUEIDENTIFIER NULL,
    ResolvedRecordID       NVARCHAR(450) NULL,
    PreviousConversationID UNIQUEIDENTIFIER NULL,
    CONSTRAINT FK_Conversation_ResolvedEntity
        FOREIGN KEY (ResolvedEntityID) REFERENCES ${flyway:defaultSchema}.Entity(ID),
    CONSTRAINT FK_Conversation_PreviousConversation
        FOREIGN KEY (PreviousConversationID) REFERENCES ${flyway:defaultSchema}.Conversation(ID);
-- Non-clustered index on (ResolvedEntityID, ResolvedRecordID) and on VisitorKey.
-- RecordID intentionally NOT an FK (polymorphic; supports composite/non-uuid PKs).
```

**`MJ: AI Agent Notes`** (entity-agnostic memory scope, R5):
```sql
ALTER TABLE ${flyway:defaultSchema}.AIAgentNote ADD
    ScopeEntityID  UNIQUEIDENTIFIER NULL,
    ScopeRecordID  NVARCHAR(450) NULL,
    CONSTRAINT FK_AIAgentNote_ScopeEntity
        FOREIGN KEY (ScopeEntityID) REFERENCES ${flyway:defaultSchema}.Entity(ID);
-- UserID retained for back-compat. When Scope* is set, the note is scoped to that record.
```

**`MJ: Widget Instances`** (the toggle, R6):
```sql
ALTER TABLE ${flyway:defaultSchema}.WidgetInstance ADD
    RememberReturningVisitors BIT NOT NULL DEFAULT 0,
    VisitorMemoryRetentionDays INT NULL;   -- NULL = use system default
```

> Per migration rules: one `ALTER TABLE` per table with comma-separated `ADD`s; `sp_addextendedproperty` for every new column; no `__mj_*` columns; no manual FK indexes (CodeGen emits `IDX_AUTO_MJ_FKEY_*`). The `(ResolvedEntityID, ResolvedRecordID)` and `VisitorKey` lookup indexes are *not* FK indexes, so author them by hand. **Write code against the new fields only after the migration + CodeGen have run** (CLAUDE rule 2b — no `.Get()/.Set()` stand-ins).

## 4. Phased task breakdown

### Phase RV0 — Schema + CodeGen (foundation)
- [ ] Migration: the three `ALTER TABLE`s above + extended properties + the two non-FK lookup indexes.
- [ ] Run migration + CodeGen on a clean DB; verify generated entity types carry the new fields. Sync metadata.
- [ ] **Acceptance:** `ConversationEntity.ResolvedEntityID/ResolvedRecordID/PreviousConversationID/VisitorKey`, `MJAIAgentNoteEntity.ScopeEntityID/ScopeRecordID`, and `WidgetInstance.RememberReturningVisitors` exist as strongly-typed properties.

### Phase RV1 — Durable visitor anchor (the cookie)
- [ ] Widget mints a durable, opaque `VisitorKey` on first visit (separate from the ephemeral guest JWT / per-session `ExternalID`); store as a long-lived first-party cookie. **Gated on `RememberReturningVisitors`** — no cookie when off.
- [ ] On widget load with `RememberReturningVisitors=1` and a cookie present, send the `VisitorKey` to the mint endpoint; server stamps it on the new `Conversation` and looks up the most recent prior conversation for that key to set `PreviousConversationID`.
- [ ] **Acceptance:** a returning cookie produces a new conversation whose `VisitorKey` matches and whose `PreviousConversationID` points at the prior one; with the toggle off, no cookie is set and no link is made.

### Phase RV2 — Recap on close (reuse Memory Manager)
- [ ] On session/conversation close (toggle on), summarize the conversation into an **Active** auto-generated `AI Agent Note`, scoped via the polymorphic pair when resolved, else carrying the `VisitorKey` for the unresolved case. Reuse the existing Memory Manager summarization — do not hand-roll a summarizer.
- [ ] Respect `VisitorMemoryRetentionDays` (or system default) for decay/archival.
- [ ] **Acceptance:** after a guest conversation closes, exactly one Active recap note exists, source-linked to the conversation; it decays/archives per retention.

### Phase RV3 — Inject on return
- [ ] At realtime session prep / agent dispatch, resolve "this conversation's identity" → relevant recap notes (by `Scope*` pair when resolved; by `VisitorKey`-linked prior conversations when anonymous) and feed them through the **existing** memory injection.
- [ ] **Acceptance:** a returning visitor's agent opens with awareness of the prior conversation; a brand-new visitor gets none; the injection path is the existing one (no bespoke prompt stuffing).

### Phase RV4 — Identity resolution + merge
- [ ] On magic-link verify / host-passed identity, set `Resolved(EntityID, RecordID)` on the live conversation and back-fill the visitor's prior conversations found by `VisitorKey`; merge visitor-scoped memory onto the resolved pair.
- [ ] Resolution target is deployment-configurable (e.g. `__mj.User`, `__mj_BizAppsCommon.Person`) — the resolver returns an `(EntityID, RecordID)` pair, never a hardcoded User.
- [ ] **Acceptance:** an anonymous visitor who verifies has their prior anonymous context attributed to the resolved record; a second post-verification visit pulls memory via the resolved pair, not the cookie.

### Phase RV5 — Privacy controls
- [ ] Visitor-facing notice when remembering is on; a "forget me" that archives the visitor's memory notes and clears the cookie/`VisitorKey` linkage.
- [ ] **Acceptance:** "forget me" leaves no Active memory and no resolvable cookie linkage for that visitor.

## 5. Anti-patterns (do not)

- ❌ **Overload `LastSessionID`/`RootLastSessionID`** for returning visits — they are resume-scoped and the replay viewer walks them. Use `Conversation.PreviousConversationID` (R2).
- ❌ **A `Conversation.UserID`-only resolved identity** — false premise (R1). Always the polymorphic pair.
- ❌ **An `FKEY` on `RecordID`** — it is polymorphic across entities; index it, don't constrain it.
- ❌ **A new recap table or a second injection path** — recaps are agent memory notes through the existing Memory Manager + injection (R4).
- ❌ **A tracking cookie or stored recap when the toggle is off** — default off, opt-in only (R6).
- ❌ Writing TS against the new columns before migration + CodeGen run (CLAUDE rule 2b).

---

## 6. Open questions / smaller-scope fallback

- **Memory scoping (R5) is the one place that ripples beyond the conversation.** If we want returning-visitor memory shipped *without* touching the notes schema, the fallback is: leave `AI Agent Notes` as-is and have the injection resolver map "this conversation's resolved `(EntityID, RecordID)` (or `VisitorKey`) → relevant notes" at prep time, keeping the polymorphic pointer only on the conversation. Smaller change; the cost is that agent memory stays User-centric internally. Recommend R5 (entity-agnostic notes) long-term; the conversation-level pair is identical either way.
- **Visitor aggregation across widgets/devices:** a normalized `Widget Visitor` entity (key, widget, resolved pair, first/last-seen) would let one resolved identity span multiple cookies/devices. Deferred — `VisitorKey` on the conversation covers the single-cookie case without a new entity.
- **Telephony/meeting parity:** the same model applies (a phone number / meeting participant resolves to a record), but the anchor differs (caller DID vs. cookie). Out of scope for the first cut; the conversation-level fields are anchor-agnostic, so it extends cleanly.
