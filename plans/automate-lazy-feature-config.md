# Plan: Automate Lazy Feature Config Generation

## Problem

`lazy-feature-config.ts` is a **manually maintained** mapping of 78+ resource type strings to dynamic `import()` loaders. Every time a new dashboard component is added with `@RegisterClass(BaseResourceComponent, 'SomeResource')`, a developer must:

1. Add the component to the correct feature module
2. Manually add an entry to `lazy-feature-config.ts`
3. Know which loader function corresponds to which subpath export

This is error-prone (the Playwright testing in PR #2212 failed initially because the new components weren't registered here) and inconsistent with MJ's philosophy of **fully automated metadata-driven code generation**. The manifest system already auto-generates eager class registrations — lazy registrations should work the same way.

## Why This Matters

The manifest generator (`mj codegen manifest`) already has **all the metadata needed** to generate this config:

| What the generator knows | How it maps to lazy config |
|---|---|
| `@RegisterClass(BaseResourceComponent, 'AIModelsResource')` | The **key** in the config |
| Package: `@memberjunction/ng-dashboards` | Which package to import from |
| Source file: `src/AI/components/models/...` | Which feature module it belongs to |
| `package.json` subpath exports: `./ai-dashboards.module` | The **import path** for the loader |

The generator already walks the dependency tree, parses AST for decorators, and verifies public API exports. Generating the lazy config is a natural extension.

## Proposed Solution

### Extend `mj codegen manifest` with a `--lazy-config` flag

```bash
mj codegen manifest \
  --output ./src/generated/class-registrations-manifest.ts \
  --lazy-config ./src/generated/lazy-feature-config.generated.ts \
  --lazy-base-classes BaseResourceComponent,BaseDashboard \
  --appDir ./packages/MJExplorer
```

### How It Works

#### Step 1: Identify Lazy-Loadable Classes

During the existing AST scan, when the generator finds a `@RegisterClass` decorator:

```typescript
@RegisterClass(BaseResourceComponent, 'VectorManagementResource')
export class VectorManagementResourceComponent extends BaseResourceComponent { ... }
```

It already extracts:
- `baseClassName`: `BaseResourceComponent`
- `key`: `'VectorManagementResource'`
- `packageName`: `@memberjunction/ng-dashboards`
- `filePath`: `src/AI/components/vectors/vector-management-resource.component.ts`

If `baseClassName` is in the `--lazy-base-classes` list, mark this class as lazy-loadable.

#### Step 2: Resolve Subpath Export

For each lazy-loadable class, determine which subpath export contains it:

1. Read the package's `package.json` `exports` field
2. For each subpath (e.g., `./ai-dashboards.module`):
   - Read its `.d.ts` entry point
   - Follow `export *` chains to build the set of exported names
   - Check if our class is in that set
3. Map: `VectorManagementResourceComponent` → subpath `./ai-dashboards.module`

This groups all classes by their chunk:
```
ai-dashboards.module → [AIModelsResource, AIPromptsResource, VectorManagementResource, ...]
actions-dashboards.module → [ActionsOverviewResource, ActionCategoriesResource, ...]
```

#### Step 3: Generate the Config File

Output `lazy-feature-config.generated.ts`:

```typescript
/**
 * AUTO-GENERATED by mj codegen manifest --lazy-config
 * DO NOT EDIT — regenerate with: npm run mj:manifest:explorer
 *
 * Maps @RegisterClass keys to their lazy-loading chunks.
 * Generated from @RegisterClass(BaseResourceComponent, 'key') decorators
 * found across the dependency tree.
 */

function featureLoader(importFn: () => Promise<unknown>): () => Promise<void> {
  return () => importFn().then(() => {});
}

// ── @memberjunction/ng-dashboards ──
const loadAI = featureLoader(() => import('@memberjunction/ng-dashboards/ai-dashboards.module'));
const loadActions = featureLoader(() => import('@memberjunction/ng-dashboards/actions-dashboards.module'));
const loadTesting = featureLoader(() => import('@memberjunction/ng-dashboards/testing-dashboards.module'));
// ... one loader per unique subpath

export const LAZY_FEATURE_CONFIG: Record<string, () => Promise<void>> = {
  // @memberjunction/ng-dashboards → ai-dashboards.module (12 classes)
  'AIModelsResource': loadAI,
  'AIPromptsResource': loadAI,
  'AIAgentsResource': loadAI,
  'AIMonitorResource': loadAI,
  'AIConfigResource': loadAI,
  'AIAgentRequestsResource': loadAI,
  'VectorManagementResource': loadAI,
  'DuplicateDetectionResource': loadAI,
  'AutotaggingPipelineResource': loadAI,
  // ...

  // @memberjunction/ng-dashboards → actions-dashboards.module (7 classes)
  'ActionsOverviewResource': loadActions,
  'ActionCategoriesResource': loadActions,
  // ...
};

export const LAZY_FEATURE_CONFIG_COUNT = 78;
export const LAZY_FEATURE_CONFIG_GENERATED = true;
```

#### Step 4: Integrate into Build Pipeline

Add to `mj:manifest:explorer` script (already exists):

```json
{
  "mj:manifest:explorer": "mj codegen manifest --output ... --lazy-config src/generated/lazy-feature-config.generated.ts --lazy-base-classes BaseResourceComponent"
}
```

The existing `app.module.ts` import changes from:
```typescript
// Before (manual)
import { LAZY_FEATURE_CONFIG } from '../services/lazy-feature-config';

// After (generated)
import { LAZY_FEATURE_CONFIG } from '../generated/lazy-feature-config.generated';
```

### Edge Cases

**Multiple packages with lazy chunks**: The current manual config already spans multiple packages — `@memberjunction/ng-dashboards` (11 subpath chunks) and `@memberjunction/ng-explorer-settings` (separate package with its own loader). The generator already walks the full dependency tree, so any package with subpath exports containing `BaseResourceComponent` classes automatically gets its own loader group. This is fully generic — not hardcoded to dashboards. Third-party or customer packages that follow the same pattern (subpath exports + `@RegisterClass`) would be picked up automatically.

**New packages added later**: As long as the new package is a dependency of MJExplorer and has subpath exports in `package.json`, the generator picks it up automatically. Zero manual config needed.

**Classes in the main bundle (no subpath export)**: If a `@RegisterClass(BaseResourceComponent, 'Foo')` class is exported from the package's root entry point (not a subpath), it's already in the eager manifest and doesn't need a lazy entry. The generator should skip these.

**Collision detection**: If two subpaths export the same key string, emit a build-time error (same as the manifest generator does for class name collisions).

### Migration

1. Generate the lazy config: `npm run mj:manifest:explorer`
2. Diff the generated output against the manual `lazy-feature-config.ts` — should be identical
3. Delete the manual file, update the import
4. Verify with `npm run build` in MJExplorer
5. Run Playwright smoke test to confirm lazy loading still works

### Result

After this change:
- Adding a new dashboard component requires **zero manual config** — just `@RegisterClass` + export from the feature module
- `mj codegen manifest` handles both eager and lazy registrations
- Consistent with MJ's fully automated metadata-driven approach
- One less file for developers to forget to update
