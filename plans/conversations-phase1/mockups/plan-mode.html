Plan Mode

A per-request toggle (default off) — when on, a plan-capable agent returns an editable plan for the user to approve, edit, or reject before it executes. Reuses MJ's AIAgentRequest human-in-the-loop approval.

What: The Plan-first switch lives on the agent pill itself inside the composer, so it’s scoped to exactly the agent you @mentioned. The returned plan is an inline, editable checklist card in the thread with drag-to-reorder, click-to-edit steps, and Approve / Edit / Reject (affirmative on the LEFT per MJ convention).

Why world-class: Zero new chrome — it extends the mention model users already know. The decision (“plan or just go”) sits right next to the subject (the agent), and the plan stays in conversational flow so it reads like the rest of the thread.

Tradeoff: The toggle is small and per-mention; users who want plan mode “always on” for an agent must re-toggle each request, and a busy composer can hide the affordance.

What: A persistent “Plan mode” switch on the composer (default off). With it on, the send button reads Review plan and submitting opens a focused modal that surfaces the full multi-step plan with drag handles, up/down reorder, per-step remove, add-step, and Approve / Reject.

Why world-class: A dedicated modal gives the plan room to breathe and signals a real decision point — ideal for high-stakes, destructive, or many-step work. The switch is sticky for the session, so a user doing a batch of careful operations sets it once.

Tradeoff: A modal interrupts conversational flow and adds a click; the persistent switch can be left on and surprise a user who expected immediate execution.

Before approval the panel is fully editable and the footer reads Run / Discard; after pressing Run the same steps animate through done → active → pending.

What: A Claude-Code-style side panel docks beside the conversation. The plan appears there for review and editing (reorder, edit, add steps); pressing Run approves it, and the same panel then shows live step-by-step progress (done / active-with-progress / pending) while the chat stays clean.

Why world-class: Separates “the plan” from “the chatter” — the durable artifact gets a persistent home that doubles as the execution dashboard. Plan and progress share one surface, so the user’s mental model is continuous from approval through completion.

Tradeoff: Costs horizontal space and added layout complexity; on narrow/mobile widths the side canvas must collapse to a drawer, and the richer surface is overkill for a one-line plan.
