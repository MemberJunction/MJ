MemberJunction Mobile · Phase 1 Visual Spec

Implementation handoff. Every screen, full resolution, with notes.

Phase 1 of the MJ mobile app. Native iOS & Android via React Native (Expo). Chat is primary — a first-class multi-agent conversation experience with artifacts, designed-for voice. Data Explorer is read-only secondary. This page is the visual contract: the architecture plan, design tokens, data sources, and interaction notes live in README.md. Deviations should be discussed and reflected back into both.

The "Ask Skip" bridge: every Explorer surface ends with a CTA that opens a new conversation seeded with the current context (record reference, query name + params, dashboard reference). This is the bridge that makes Data Explorer relevant on mobile — without it, a phone-sized record view is just a worse desktop view. With it, the user lands on a record, sees something interesting, and immediately can have a conversation about it.

For the implementing agent

Mockups are the design contract. Pixel match isn't required, but proportions, spacing, color tokens, and interaction patterns are. Use the same design tokens listed in README.md §4.1 as React Native theme constants from day one. Build the chat surface first — it's the differentiator and the highest-value workflow. Don't ship Data Explorer until chat feels great.

The artifact dock collapsed-handle pattern (Part 4.2 §B) is one of the more subtle pieces — make sure that lands. It's the difference between artifacts feeling like first-class citizens and them feeling like a distraction.

For artifact rendering, read README §4.3 first. Two architectural decisions there worth flagging:

  • Interactive Component artifacts reuse @memberjunction/react-runtime directly — that package is platform-agnostic by design; we don't need the Angular bridge (ng-react). Phase 1 ships a "View on desktop" stub; Phase 2 wires the full runtime with an RN adapter for ReactRootManager.
  • Markdown is not a stock library install. @memberjunction/ng-markdown ships Prism, Mermaid, and custom extensions (code-copy, collapsible headings, SVG). We extract a new @memberjunction/markdown-core that owns parsing + extensions + AST output; the Angular and RN renderers become thin consumers. Avoids the divergence trap.
