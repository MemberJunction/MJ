# MJExplorer Bundle Optimization: Lazy Loading & Dynamic Imports

## Context

MJExplorer's `main.js` is **15.2 MB uncompressed** with zero route-level code splitting. The root cause is twofold:
1. `MJExplorerModulesBundle` eagerly imports all MJ + Kendo modules
2. The **ng-bootstrap manifest** (`mj-class-registrations.ts`) has static imports from `@memberjunction/ng-dashboards` (58 classes), which forces ESBuild to include the entire 2.9 MB package in the initial chunk ‚Äî even if the DashboardsModule itself were lazy-loaded

This plan covers:
- **Goal 1**: Lazy-load dashboard feature modules (~4.9 MB savings)
- **Goal 2**: Dynamic import of exceljs (934 KB) and dhtmlx-gantt (601 KB)
- **Execution**: All coding done by Claude worker in Docker workbench on a `bundle-optimization` feature branch

---

## Architecture: Why Standard `loadChildren` Won't Work

MJExplorer does NOT use Angular's router for component resolution. All routes resolve to `SingleRecordComponent`, then `ResourceContainerComponent.loadComponent()` (line 53-95 in `packages/Angular/Explorer/explorer-core/src/lib/generic/resource-container-component.ts`) calls:

```typescript
const resourceReg = MJGlobal.Instance.ClassFactory.GetRegistration(BaseResourceComponent, this.Data.ResourceType);
```

Components must be registered in ClassFactory BEFORE this lookup. The solution: **add a lazy-loading fallback path** ‚Äî when `GetRegistration()` returns null, dynamically import the chunk containing that resource type (which fires `@RegisterClass` side effects), then retry the lookup.

## Implementation Plan

### Phase 1: ng-bootstrap-lite Manifest Variant (Unblocks Everything Else)

**Problem**: `packages/Angular/Bootstrap/src/generated/mj-class-registrations.ts` is AUTO-GENERATED by `mj codegen manifest`. It statically imports 58 classes from `@memberjunction/ng-dashboards` and 6 from `@memberjunction/ng-explorer-settings`. ESBuild cannot code-split away these imports regardless of Angular module structure.

**Solution**: Create an `ng-bootstrap-lite` package following the **existing `ServerBootstrapLite` pattern**.

The codebase already has this pattern:
- `packages/ServerBootstrapLite/` uses `--exclude-packages` flags to generate a lite manifest
- NPM script `mj:manifest:server-bootstrap-lite` runs the generator with exclusions
- This survives regeneration because it's a separate generator invocation, not a manual edit

#### What to create:

1. **New package: `packages/Angular/BootstrapLite/`**
   - `package.json` ‚Äî depends on all ng-bootstrap dependencies EXCEPT `@memberjunction/ng-dashboards` and `@memberjunction/ng-explorer-settings`
   - `src/generated/mj-class-registrations.ts` ‚Äî auto-generated lite manifest
   - `src/public-api.ts` ‚Äî re-exports manifest + anything else from ng-bootstrap that isn't lazy

2. **New npm script in root `package.json`:**
   ```bash
   "mj:manifest:ng-bootstrap-lite": "mj codegen manifest --appDir ./packages/Angular/BootstrapLite --exclude-packages @memberjunction/ng-dashboards --exclude-packages @memberjunction/ng-explorer-settings --output ./packages/Angular/BootstrapLite/src/generated/mj-class-registrations.ts --scan-dist"
   ```

3. **Update `mj:manifest` script** to include the new lite variant in the full manifest generation run.

#### Backward compatibility:
- The existing `@memberjunction/ng-bootstrap` package and its manifest are **completely unchanged**
- Non-MJExplorer consumers continue using the full manifest
- MJExplorer switches to importing `@memberjunction/ng-bootstrap-lite`

### Phase 3: Lazy Loading Infrastructure in explorer-core

#### 3A. Create `LazyModuleRegistry` service

**File**: `packages/Angular/Explorer/explorer-core/src/lib/services/lazy-module-registry.ts` (NEW)

```typescript
@Injectable({ providedIn: 'root' })
export class LazyModuleRegistry {
  private registry = new Map<string, () => Promise<void>>();
  private loadedChunks = new Set<string>();
  private pendingLoads = new Map<string, Promise<void>>();

  Register(resourceType: string, loader: () => Promise<void>): void { ... }
  RegisterBulk(mappings: Record<string, () => Promise<void>>): void { ... }
  async Load(resourceType: string): Promise<boolean> { ... }  // true if found & loaded
}
```

#### 3B. Modify `ResourceContainerComponent.loadComponent()`

**File**: `packages/Angular/Explorer/explorer-core/src/lib/generic/resource-container-component.ts`

Change `loadComponent()` from sync to async. Add lazy loading fallback:

```typescript
async loadComponent() {
  try {
    this._loaded = true;
    let resourceReg = MJGlobal.Instance.ClassFactory.GetRegistration(
      BaseResourceComponent, this.Data.ResourceType
    );

    if (!resourceReg) {
      // Try lazy loading the feature module containing this resource type
      const lazyRegistry = inject(LazyModuleRegistry);
      const loaded = await lazyRegistry.Load(this.Data.ResourceType);
      if (loaded) {
        resourceReg = MJGlobal.Instance.ClassFactory.GetRegistration(
          BaseResourceComponent, this.Data.ResourceType
        );
      }
    }

    if (!resourceReg) {
      throw new Error(`Unable to find resource registration for ${this.Data.ResourceType}`);
    }
    // ... rest of existing component creation logic unchanged
  }
}
```

Also update `ngOnChanges` to `await` the now-async `loadComponent()`.

#### 3C. Create Lazy Feature Registry Configuration

**File**: `packages/Angular/Explorer/explorer-core/src/lib/services/lazy-feature-config.ts` (NEW)

Maps resource type strings to dynamic `import()` calls for each feature chunk:

#### 3D. Wire up in MJExplorer app

**File**: `packages/MJExplorer/src/app/app.module.ts`

- Import `@memberjunction/ng-bootstrap-lite` instead of `@memberjunction/ng-bootstrap`
- Import `CoreDashboardsModule` eagerly (for Home, EntityAdmin, etc.)
- In `APP_INITIALIZER`, register `LAZY_FEATURE_CONFIG` into `LazyModuleRegistry`

### Phase 4: Dynamic Import of Heavy Libraries

#### 4A. dhtmlx-gantt (601 KB) ‚Äî Simplest change

**File**: `packages/Angular/Generic/tasks/src/lib/components/gantt-task-viewer.component.ts`

Single static import: `import { gantt } from 'dhtmlx-gantt'`

**Change**:
- Remove top-level static import
- In `ngAfterViewInit()`: `const { gantt } = await import('dhtmlx-gantt')`
- Store `gantt` instance as component property for use in other methods
- Show `<mj-loading>` while library loads

Single file, single consumer, already standalone component.

#### 4B. ExcelJS (934 KB)

**File**: `packages/MJExportEngine/src/export-engine.ts`

Static `import { ExcelExporter } from './excel-exporter'` (line 10) pulls in exceljs for all consumers.

**Change**:
- Remove static import of `ExcelExporter`
- Refactor excel-specific methods to dynamically import:
  - In `export()` and `toExcel()`: the path already goes through `createExporter()` ‚Üí make it async
  - In `toExcelMultiSheet()`: directly uses `new ExcelExporter()` ‚Üí use `await import('./excel-exporter')`
  - `getMimeType('excel')` and `getFileExtension('excel')`: return hardcoded `'application/vnd.openxmlformats-officedocument.spreadsheetml.sheet'` and `'xlsx'` without needing ExcelExporter
- Keep `createExporter()` sync for csv/json, add `createExporterAsync()` that handles all formats
- The public async methods (`export`, `toExcel`, `toCSV`, `toJSON`, `toExcelMultiSheet`) call the async path internally

**File**: `packages/MJExportEngine/src/excel-exporter.ts` ‚Äî no changes needed

#### 4C. AG Grid (1.1 MB) ‚Äî DEFERRED

AG Grid is used on virtually every entity view page. Lazy loading would add latency to the most common user flow. Defer to a future phase.

## Risk Mitigation

| Risk | Mitigation |
|------|-----------|
| Lazy chunk loading delay | `ResourceContainerComponent` shows `<mj-loading>` during dynamic import |
| NgModule injector scope for lazy modules | Dashboard services already use `providedIn: 'root'` ‚Äî no change needed |
| Feature module missing a Kendo import | Build will fail early ‚Äî each module must declare its own imports |
| Backward compatibility for non-MJExplorer consumers | Original `DashboardsModule` and `mj-class-registrations.ts` unchanged |
| ExcelJS sync callers of `createExporter('excel')` | Keep sync method for csv/json; add async variant; public methods already async |
| Manifest regeneration overwrites changes | No manual edits to generated files ‚Äî we create a new lite package with its own generator invocation |
