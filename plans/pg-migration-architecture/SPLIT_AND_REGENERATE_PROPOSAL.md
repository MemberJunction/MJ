# Split-and-Regenerate: a deterministic PG migration path

**Status:** Proposal ‚Äî for review by the build engineer, Amith, and whoever owns the release pipeline.
**Supersedes:** the conversion strategy in `ARCHITECTURE_PLAN.md` for *new* migrations. Does **not** touch already-committed `.pg.sql` files.
**One-line thesis:** Stop regex-translating the 80‚Äì100% of every migration that a machine generated. Regenerate it natively on PG instead, and translate only the small, regular hand-written remainder.

---

## 1. The problem, stated honestly

The `/pg-migrate` skill is a six-phase orchestration that spins up Docker, runs Claude Code *inside* the container with `--dangerously-skip-permissions`, polls for completion markers, and drives a regex-based converter over the migration backlog. It is slow, hard to reason about, and brittle. Converting the backlog is a build-engineer task done in painful batch runs ‚Äî not part of the dev loop. **Devs only ever write SQL Server migrations.**

The brittleness is not in the orchestration. It is in the engine:

- `@memberjunction/SQLConverter` is **100% regex + string manipulation**. No AST. 28 rules plus a `PostProcessor` with 40+ regex passes acting as a band-aid for what the rules miss.
- A real AST transpiler exists in the repo ‚Äî `SQLGlotTS` (wraps Python `sqlglot`) ‚Äî but **every rule sets `BypassSqlglot = true`**. The AST path is effectively dead code. (Note: `ARCHITECTURE_PLAN.md` ¬ß2.2 claims "SQLGlot is the transpiler, rules supplement it." That is aspirational; the shipped reality is the inverse.)
- The brittlest rules ‚Äî `ProcedureToFunctionRule`, `DeclareDmlBlockRule`, `ExecBlockRule`, `TriggerRule`, `CatalogViewRule` ‚Äî exist solely to translate PL/pgSQL control flow. **Every line they touch is CodeGen output.** We are regex-translating a machine's deterministic output.

## 2. The key fact: migrations are ~80‚Äì100% machine-generated, and the split is deterministic

Migration content comes from **two generators**, each with a fixed output grammar, plus a small hand-written remainder:

- **CodeGen** emits SQL objects (views, `spCreate`/`spUpdate`/`spDelete`, view-permission GRANTs, FK indexes, `sp_addextendedproperty`). Every generated item is banner-marked:
  ```
  -- This was generated by the MemberJunction CodeGen tool.
  ```
  A minority of files additionally carry an explicit run-level delimiter (`-- CODE GEN RUN (appended from ‚Ä¶)`). **The per-item banner is the reliable boundary marker; the run delimiter is not** ‚Äî only 39 of 144 versioned migrations have it.
- **mj-sync** emits metadata seed as `DECLARE @<col>_<8hex> ‚Ä¶; ‚Ä¶ EXEC __mj.spCreate/spUpdate ‚Ä¶` blocks (the `_da319a9d`-style hashed local-variable suffix is its signature). These are the large `*_Metadata_Sync` files.
- **Hand-written** SQL: regular DDL (`CREATE TABLE`/`ALTER`/`INDEX`/constraints/seed `INSERT`s) and, rarely, dev-authored procedural SQL (`CREATE PROCEDURE`/`FUNCTION`/`TRIGGER`).

### Measured: the `MigrationSplitter` census over the real backlog

Reproduce with `npx tsx packages/SQLConverter/scripts/migration-census.mts`. Across the **144 versioned (V\*) migrations** (4 baselines excluded ‚Äî they are squashed snapshots with committed `.pg.sql` counterparts, not part of the go-forward flow):

**Routing**
| Route | Files | % | Treatment |
|---|---:|---:|---|
| `transpile-only` | 84 | 58% | regular DDL ‚Äî fully automatable |
| `transpile-plus-reseed` | 49 | 34% | mj-sync metadata ‚Äî re-seed / narrow-transpile |
| `needs-hand-authoring` | 11 | 8% | hand-written procedural SQL ‚Äî human `.pg.sql` |

**Line accounting, by treatment**
| Treatment | Lines | % of all migration content |
|---|---:|---:|
| Regenerated natively (CodeGen block, discarded on PG) | 437,087 | 15.3% |
| Re-seeded via mj-sync (metadata blocks) | 2,378,535 | 83.2% |
| **General sqlglot translation surface (regular DDL)** | **41,499** | **1.5%** |

**The headline: the surface that needs general T-SQL‚ÜíPG transpilation is ~1.5% of all migration content** (~290 lines per file). 83% is mj-sync metadata and 15% is CodeGen objects ‚Äî both *regenerated by the tool that produced them*, never transpiled. The build engineer is currently pushing **100% of it** ‚Äî including six-figure-line pure-regeneration files like `Regenerate_Stored_Procs_For_Tolerant_Signatures` (81k lines, ~0% hand-written) ‚Äî through a regex gauntlet. That is the source of the sleepless nights, and it is almost entirely avoidable.

The regex burden drops from **2.86M lines ‚Üí 56K lines (2%)**, with zero error/TODO markers on what remains.

## 3. The capability that makes this work

`CodeGenLib` already has a native PG path:

- `sql_codegen.ts:53` resolves a CodeGen provider from `MJGlobal.ClassFactory` keyed on `configInfo.dbPlatform`. `PostgreSQLCodeGenProvider` is registered alongside `SQLServerCodeGenProvider`.
- The PG provider has genuine dialect-aware logic: per-entity serialized execution (`sql_codegen.ts:203‚Äì216`), PG view-refresh handling (`:982`), self-FK view-only NameField handling (`:1747‚Äì1795`), plus 42P16 view-dependency capture/restore (`providers/postgresql/viewDependencyCapture.ts`, `viewFallback.ts`).
- `Misc/sql_logging.ts:42` already swaps the CodeGen SQL output folder `./migrations/v5/` ‚Üí `./migrations-pg/v5/` when `dbPlatform() === 'postgresql'`.
- `dbPlatform` is set from `DB_PLATFORM=postgresql` (`Config/config.ts:572`, accessor `:947`).

So "regenerate the CodeGen block natively on PG" is not new infrastructure ‚Äî it is exercising a path that already exists and is integration-tested (per `ARCHITECTURE_PLAN.md` ¬ß1, 59+ PG integration tests cover CodeGen sprocs/views).

This also matches the **two-pass PG install flow already documented**: `migrate` (DDL) ‚Üí `codegen` ‚Üí `migrate` (Metadata_Sync). For a fresh PG install you do not need each historical migration's *intermediate* sproc text ‚Äî you need the **end-state schema** to match SQL Server. One native CodeGen pass over the final metadata produces exactly that, deterministically.

## 4. Proposed architecture: four content categories per migration

Split each SS migration deterministically. The boundary between hand-authored content and the CodeGen block is found by locating the first generated-item banner and walking up to its `/*` item-opener (or the explicit `-- CODE GEN RUN` line when present). The hand-authored region is then classified by generator grammar. This is implemented and validated ‚Äî see `packages/SQLConverter/src/MigrationSplitter.ts` (`splitMigration`).

| # | Category | Content | Treatment | General-transpile surface |
|---|---|---|---|---|
| **A** | CodeGen block | views, sprocs, view-permission GRANTs, FK indexes, extended properties (banner-marked) | **Discard.** Regenerated by a single native `mj codegen` PG pass after DDL is applied. | **none** |
| **M** | mj-sync metadata | `DECLARE @<col>_<hash> ‚Ä¶; ‚Ä¶ EXEC spCreate/spUpdate ‚Ä¶` seed blocks | **Re-seed** via `mj sync push` against PG, *or* narrow-transpile the fixed mj-sync grammar (`DECLARE @x`‚Üí`DO $$ DECLARE`, `EXEC spX`‚Üí`PERFORM "spX"`). Either way it is the generator's fixed grammar, not arbitrary T-SQL. | **none / narrow & fixed** |
| **B** | Regular hand-DDL | `CREATE TABLE`, `ALTER TABLE ADD/ALTER COLUMN`, `CREATE INDEX`, FK/CHECK/UNIQUE constraints, plain seed `INSERT`/`UPDATE` | AST transpile via `SQLGlotTS` (activate the dead path) + a thin MJ shim: `${flyway:defaultSchema}` quoting, `BIT`‚Üí`boolean`, `GETUTCDATE()`, idempotency wrappers | minimal, **structural only** (~1.5% of content) |
| **C** | Hand-written procedural SQL | dev-authored `CREATE PROCEDURE`/`FUNCTION`/`TRIGGER` | **Detect and flag for a human-authored `.pg.sql`.** Do not regex-translate. | **none ‚Äî escalated to human** |

The genuinely-hard PL/pgSQL translation (category C) is what regex was always going to fail at ‚Äî but the census found it is exactly **11 of 144 versioned migrations**, all enumerable up front:

```

V202604261352 Scoped_EntityField_SPs                               CREATE PROC
V202605091143 Add_IsComputed_To_EntityField                        CREATE PROC
V202605281538 Fix_AllowUpdateAPI_On_Virtual_Transition             CREATE PROC
V202606022336 Scheduling_Engine_Atomic_Sprocs                      CREATE PROCEDURE
V202606040230 AgentRunWatchdog_Maintenance_Sprocs                  CREATE PROCEDURE
```

(Plus the 4 baselines, which contain hand-written triggers/functions but already have committed `.pg.sql` counterparts.) The tool flags these with a "needs hand-authoring" report entry instead of silently mangling them.

### The new pipeline (replaces the 6-phase skill)

A plain, deterministic script ‚Äî no Docker-in-Docker, no nested Claude, no completion-marker polling:

```
for each new SS migration (those without a .pg.sql counterpart):
  1. splitMigration() ‚Üí {handAuthored, codeGenBlock, regions, routing}
  2. routing === needs-hand-authoring ‚Üí record in NEEDS_HAND_AUTHORING report, skip file
  3. transpile the regular-DDL part of handAuthored (sqlglot + MJ shim);
     keep mj-sync metadata blocks (category M) for re-seed/narrow-transpile;
     DROP the codeGenBlock (category A ‚Äî regenerated below).
     ‚Üí write migrations-pg/v5/<name>.pg.sql
assemble + verify (once, not per-file):
  4. fresh PG db ‚Üí bootstrap cdp_* roles ‚Üí mj migrate (applies all .pg.sql DDL in order)
  5. mj codegen  (DB_PLATFORM=postgresql)  ‚Üí regenerates ALL SQL objects natively into PG
  6. seed metadata: mj sync push (preferred) OR apply narrow-transpiled M blocks
  7. schema + row-count diff vs a fresh SQL Server build of the same tag ‚Üí parity report
```

Step 5 is the linchpin: every view/sproc/permission that category A would have translated is instead generated correctly by construction. Step 6 does the same for metadata (category M) ‚Äî regenerate the seed via the generator, don't transpile it.

## 5. What this deletes / demotes

- **Delete** (or quarantine as legacy-only, used solely to re-read already-committed files): `ProcedureToFunctionRule`, `FunctionRule`, `TriggerRule`, `CatalogViewRule`, `GrantRule`, `ExtendedPropertyRule`, `SequenceDeduplicator`. These all target category-A generated content, now regenerated natively.
- **Demote to category-M fallback only**: `DeclareDmlBlockRule`, `ExecBlockRule`. If mj-sync re-seed (step 6) covers everything, these are deleted too; if a narrow-transpile fallback is needed, they survive scoped to the fixed mj-sync grammar.
- **Keep, but narrowed to category B**: `CreateTableRule`, `AlterTableRule`, `CreateIndexRule`, `InsertRule`, `ConditionalDDLRule`, and the regular-DDL slice of `ExpressionHelpers`/`TypeResolver`/`PostProcessor`.
- **Activate**: `SQLGlotTS.transpile()` for category-B expressions/DDL ‚Äî the AST path the current rules bypass.
- **Replace**: the `/pg-migrate` skill's six phases + Docker orchestration ‚Üí the 7-step script above. (Phase 4/4b full-stack smoke + CRUD tests can stay as an optional final gate; they're orthogonal and valuable.)

## 6. Migration / rollout

1. **Freeze the past.** All committed `migrations-pg/v5/*.pg.sql` stay byte-for-byte as-is. They are a deployed historical ledger; re-converting them is a breaking change (existing rule #6 in the skill). The new flow applies **only to new migrations** going forward.
2. **Parallel-run for confidence.** For the next N new migrations, produce `.pg.sql` both the old way and the new way; schema-diff both results against SQL Server. Promote the new flow once parity holds (expect it to be *better* ‚Äî category A objects are now generated, not translated).
3. **Cut over** the skill once parallel-run is clean for a release cycle.

## 6.6 Live-PG validation results

Built PG_A (known-good oracle) by applying the committed `migrations-pg/v5` via Skyway to a fresh `postgres-claude` DB, snapshotted it, then ran `mj codegen --skipfiles` against an intact clone and diffed:

```
                tables   cols   constraints   views   column-comments   routines
known-good (A)    326    3880      3916        334        2309           1354
after codegen     326    3880      3916        334        2309           1716  (+362)
diff               0       0         0           0          0           +362 (all legit CodeGen fns)
```

**Findings:**
1. **CodeGen runs cleanly against PostgreSQL** (324 entities, ~41s) and is **schema-preserving** ‚Äî tables, columns, constraints, views, and all 2,309 column comments are byte-identical. The +362 routines are `fn_*_get_root_id` self-FK helpers + 2 sprocs the older committed set predated ‚Äî legitimate CodeGen output, not drift. ‚Üí **Category A regenerates correctly on PG.** ‚úÖ
2. **Column descriptions must be transpiled, not dropped.** A subagent traced the data flow: on PG, CodeGen *reads* `col_description()` into `EntityField.Description` (`B202605241137‚Ä¶Baseline.pg.sql:32312`; `PostgreSQLCodeGenProvider.ts:2436`) and **never emits** `COMMENT ON COLUMN`. Drop the comments ‚Üí blank descriptions. Hence the `sp_addextendedproperty`‚Üí`COMMENT ON` transform is **required** (now implemented in the dialect).

## 7. Risks & open questions

- **üîë BIGGEST OPEN QUESTION ‚Äî mj-sync re-seed viability (83% of content rides on this):** Category M is 83.2% of all migration content. The clean answer is to re-seed it via `mj sync push` against PG rather than transpiling the `DECLARE @x; EXEC spCreate` blocks. **This works for end-state only if the `/metadata/` directory is the source of truth for everything those blocks seed.** Two things to verify: (a) every `*_Metadata_Sync` migration's content is reproducible from the current `/metadata/` tree via `mj sync push`; (b) one-off data backfills that are *not* in `/metadata/` (e.g. `Backfill_Attachment_Artifacts`) are handled ‚Äî these are plain DML (category B) and transpile fine, but must not be confused with mj-sync seed. **Fallback if re-seed is incomplete:** narrow-transpile the mj-sync grammar ‚Äî it is a *fixed* template (hashed `DECLARE @x`, `EXEC spCreate/spUpdate`), so a single focused rule covers it deterministically, unlike arbitrary T-SQL.
- **Ordering dependency (low):** Does any later hand-DDL depend on an earlier *generated* sproc/view? Generally no ‚Äî hand-DDL creates tables/columns/seed-rows; generated objects are leaf outputs that depend on tables, not vice-versa. **To verify:** scan category-B content for references to generated object names. If a real case exists, that migration falls to category C (hand-authored).
- **Category-C detection accuracy:** must be conservative ‚Äî false-negative (a procedural migration wrongly routed to B) is worse than false-positive. `MigrationSplitter` routes a file to `needs-hand-authoring` on any `CREATE [OR ALTER] PROCEDURE|PROC|FUNCTION|TRIGGER` in the comment/string-stripped hand region (generated sprocs are below the boundary and excluded). The census found 11 such files; spot-check that none are false positives and that no procedural SQL hides in a category-B/M file (e.g. an inline `DO`-block trigger body). Err toward flagging.
- **CodeGen end-state vs per-migration state:** the new flow regenerates objects from *final* metadata, so intermediate migrations no longer carry their own object text on PG. This is fine for fresh installs (the only PG scenario today) but means PG does not support "migrate to an arbitrary historical tag and have period-correct sprocs." Confirm no one relies on that for PG. (SQL Server is unaffected.)
- **sqlglot coverage of category B:** needs a test corpus of real hand-DDL headers extracted from the backlog to confirm the AST path + shim covers them without the demoted rules. Low risk (regular DDL is sqlglot's sweet spot), but must be measured before deleting rules.

## 8. Why this is the right shape

The current system tries to be a general-purpose T-SQL‚ÜíPG transpiler and fails at the hard cases (procedural SQL). But MJ doesn't *have* arbitrary T-SQL ‚Äî it has machine-generated objects (regeneratable) + regular DDL (transpilable) + a rare procedural residue (hand-authorable). Matching the tool to that reality removes the brittleness at its source instead of patching it with a 41st regex pass. The build engineer stops feeding 81,000-line generated files through string surgery, and the conversion becomes deterministic by construction for the part that matters most.
