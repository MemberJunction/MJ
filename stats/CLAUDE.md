# Repository Stats (`/stats`)

This folder tracks lines-of-code snapshots over time, with every count split into
**hand-written** vs **generated** (deterministic tool output). `README.md`, `data.csv`, and
`reports/` are fully generated by [repo-stats.mjs](repo-stats.mjs) — **do not hand-edit
them**; re-run the script instead. `analysis/` holds Claude-written narratives and is the
one place prose is authored, not generated.

## How to record a snapshot

```bash
node stats/repo-stats.mjs               # snapshot the current working tree (dated today)
node stats/repo-stats.mjs <commit>      # backfill a historical commit (dated by commit date)
node stats/repo-stats.mjs <c1> <c2> …   # backfill several commits in one run
node stats/repo-stats.mjs --render      # re-render README from data.csv without recounting
```

Requires [cloc](https://github.com/AlDanial/cloc) on the PATH (`brew install cloc`).
No npm dependencies — the script is plain Node, run it from anywhere in the repo.

Each snapshot run:

1. Counts git-tracked files only (`cloc --vcs=git --by-file`, or `cloc --git <sha> --by-file`
   for historical commits — no checkout needed), so `node_modules`, `dist`, and gitignored
   paths are excluded automatically.
2. Classifies every file as hand-written or generated via `GENERATED_PATTERNS` (see below).
3. Upserts a row into `data.csv` keyed by date (re-running on the same day replaces that
   day's row — safe and idempotent). **Backfill multiple commits in chronological order** so
   each report's "change since previous" delta compares against the right predecessor.
4. Writes a full report to `reports/YYYY-MM-DD.md`: hand/gen split by language group, a
   delta table vs. the previous snapshot, and the full per-language cloc detail.
5. Regenerates `README.md` (charts, history table, and the latest narrative analysis).

## What counts as "generated"

Classification is by repo-relative path against the `GENERATED_PATTERNS` regex list at the
top of [repo-stats.mjs](repo-stats.mjs). Current families:

| Pattern | What | Emitted by |
|---|---|---|
| `**/generated/**` | Entity classes, Angular forms, resolvers, manifests | CodeGen (output dirs per `packages/CodeGenLib/src/Config/config.ts`) |
| `**/mj-class-registrations.ts`, `**/class-registrations-manifest.ts` | Tree-shaking manifests outside `generated/` dirs | `mj codegen manifest` |
| `migrations*/​**/*Metadata_Sync*.sql` | Metadata export migrations | mj-sync |
| `migrations*/​**/B*__*.sql` | Baseline consolidation migrations | baseline tooling |
| `migrations*/​**/*CodeGen_Run*.sql` | CodeGen run SQL | CodeGen |
| `migrations-pg/**` (all) | PostgreSQL migration mirrors | pg-migrate conversion |
| `metadata/integrations/**` | Connector action metadata | generate-integration-actions |
| `packages/DBAutoDoc/test-run/**` | Committed tool-run outputs | DBAutoDoc |
| `package-lock.json` | npm lockfile | npm |

**Caveat**: classification is file-level. Hand-authored migrations that contain
CodeGen-appended sections still count as hand-written — file granularity is the practical
line. When a new generated-artifact family appears in the repo, add a pattern to
`GENERATED_PATTERNS`, update the table above, and re-run the backfill for all dates in
`data.csv` so history is classified consistently.

## Narrative analysis (`analysis/`)

Each snapshot can have a Claude-written narrative at `analysis/YYYY-MM-DD.md` explaining
*why* the numbers moved (informed by the delta table and `git log` between snapshot
commits). The script embeds the most recent analysis into `README.md` and links each one
from the history table — write the analysis file, then run `--render` to fold it in.

The `/update-stats` slash command (in `.claude/commands/`) automates the full loop:
run snapshot → study deltas and git history → write the analysis → re-render. Prefer it
over running the script manually when you want the narrative kept current.

## Conventions

- **Language buckets** are defined in `LANGUAGE_GROUPS` in the script; anything unlisted
  rolls into "Other". "Source" (the headline number) = TypeScript + JavaScript + HTML +
  CSS + Markdown — see `SOURCE_GROUPS`. SQL and JSON are tracked but reported separately
  because they're dominated by tool-emitted migrations and declarative metadata.
- **Schema changes** (new bucket or pattern) require re-running the backfill for every date
  in `data.csv` so all rows stay comparable. Get the commit for a date with
  `git rev-list -1 --before="<date> 00:00" HEAD`.
- The existing history is quarterly. Keep new backfill points on quarter boundaries unless
  there's a reason not to; ad-hoc current-tree snapshots at any time are fine.
- Snapshots are read-only over git data and touch nothing outside this folder, so the
  script is always safe to run. A full snapshot takes ~30s on the current repo.
