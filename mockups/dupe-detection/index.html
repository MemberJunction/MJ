Intelligent Duplicate Detection & Merge — UX exploration

Three divergent patterns for the LLM-reasoning layer that sits on top of MJ's existing vector duplicate-detection pipeline. Each is optimized for a different reviewer job. Same scenario, three lenses. Toggle dark mode top-right; all three are standalone and offline.

🗂️
A · Throughput

Triage Queue

Inbox-style, one match-set at a time. LLM verdict + reasoning front-and-center, keyboard-driven approve / override / skip, bulk-approve the agreeing high-confidence sets.

Best when: a human must clear hundreds of decisions fast.
⚖️
B · Precision

Diff + Reasoning Rail

Full side-by-side comparison with per-field survivor radios, plus a right rail explaining the LLM's recommendation, per-field choices, provenance posture and memory notes.

Best when: careful, field-level adjudication of a tricky set.
🗺️
C · Scale & oversight

Confidence Map

Run-level scatter of vector score × LLM confidence. Auto-merge band and disagreement quadrant called out; drag the thresholds; triage the 12 disagreements that actually need eyes.

Best when: overseeing a 100k-record run & tuning thresholds.

These aren't mutually exclusive — a likely real build is C as the run dashboard, drilling into A for bulk human review, with B as the deep-dive panel for a single tricky set. The mockups deliberately push each to its extreme so we can react to the strong forms before blending.

🧩 Grounded in the real codebase

  • Builds on DuplicateRecordDetector (vector pass, RRF, batch loop, auto-merge) — unchanged when reasoning is off.
  • New LLM fields land on MJ: Duplicate Run Detail Matches; config on MJ: Entity Documents.
  • Survivor + field choices resolve to the existing MergeRecords FieldMap (value-per-field, applied pre-merge, transactional).
  • Reasoning runs behind a pluggable seam: solo prompt by default, agent for heavy entities — both share one core instruction set, chosen per Entity Document.
