Classify is growing up

The autotagger's gaining real teeth: per-tag governance, polymorphic tag scope (so tenants only see their own taxonomy), a tag suggestions queue for ambiguous classifications, synonyms, and per-source budgets. This deck explores three visual + interaction directions for each new surface so a human reviewer can pick a direction before we wire any of it into Angular.

Six surfaces, three options each

Each option-link below opens a polished mockup in this folder. The reviewer picks A, B or C per surface (or mixes and matches).

🌳1 · Taxonomy Admin & Tree

Curating the tag hierarchy with the new governance flags (frozen, max children, max depth, requires-review) plus per-node scope and synonym editing. Today's UI is a flat panel + tree; we need to fit a lot more meaning.

OPTION A Classic split Tree on left, governance form on right. Familiar; reads dense info comfortably. OPTION B Card galaxy Subtree as a constellation of cards. Visual; better for browsing, slower to edit. OPTION C Table-first Bulk-editable grid with collapsible parents. Best for governance audits at scale.
What we need feedback on: how prominent should governance badges be on the tree itself, versus tucked behind a click? Power users will live here daily.
📥2 · Tag Suggestions Inbox

The new human-in-the-loop queue. Suggestions arrive from low-confidence classifications, frozen-subtree denials, governance failures, and Tag Health analytics (merge / low-usage / wide-node). Reviewers approve, merge into existing, or reject — sometimes by the hundred.

OPTION A Email-style inbox List + reading pane. Snappy keyboard nav. Best for triaging one at a time. OPTION B Kanban by reason Columns by Reason. Drag to approve / merge / reject. Visual prioritization. OPTION C Table + drawer Dense table with bulk actions; row click opens a drawer. Fastest for bulk merges.
What we need feedback on: two distinct workflows — single-item careful review vs. 100-item bulk merge approvals. One UI for both, or a mode toggle?
⚙️3 · ContentSource Configuration

Per-source knobs: TaxonomyMode (Constrained / AutoGrow / FreeFlow / Hybrid), TagRoot, MatchThreshold, SuggestThreshold, MaxNewTagsPerRun, MaxNewTagsPerItem, MaxTokensPerRun, MaxCostPerRun. Lots of dials to expose without overwhelming the operator.

OPTION A Stepped wizard Mode → Scope → Thresholds → Budgets. Guided; great for first-time setup. OPTION B Dense form All knobs on one page, grouped. Power-user friendly; less opinionated. OPTION C Pipeline canvas Visual flow with knobs at each stage. Beautiful; expensive to build.
What we need feedback on: threshold sliders are paired (MatchThreshold > SuggestThreshold). Should we enforce that visually, or leave it as validation?
🔒4 · Tag Scope Editor

Polymorphic scope: a tag is either Global or has rows in MJ:TagScope tying it to (Entity, Record) tuples — typically Companies / Tenants. Children must inherit. Locked when IsGlobal=1.

OPTION A Dual-pane picker Available on left, in-scope on right, transfer arrows. Classic and bulletproof. OPTION B Chip multi-picker Searchable input, chips for selected. Compact, modern; harder for 50+ scopes. Sketch in A's design notes. OPTION C Tag × Tenant matrix Heatmap view across many tags at once. Great for audits, not for single-tag edits. Sketch in A's design notes.
What we need feedback on: the inheritance rule ("child scope ⊆ parent") is hard to surface. Show it as warnings, hard-disable invalid options, or both?
🔤5 · Tag Synonyms

Alternate names that resolve to the same TagID. Source field tracks where each came from (Manual / LLM / Imported / Merged). Simple but high-volume — a tag might pick up dozens.

OPTION A Inline-edit on tag form Lives on the tag governance panel. Frictionless. Best for <20 synonyms each. OPTION B Dedicated synonyms page Browse / search across all synonyms. Useful for de-dup audits. Sketch in A's design notes. OPTION C Chip stack with bulk paste Paste-a-list dialog for bulk import. Power users with CSVs in hand. Sketch in A's design notes.
What we need feedback on: auto-generated synonyms (Source='LLM') need approval before they count. Inline review or batched on the dedicated page?
📊6 · Tag Health Dashboard

Surfaces three signals from TagHealthJob: merge candidates (high-cosine pairs that co-occur), low-usage deprecation candidates, and wide-node alerts (parents with too many direct children).

OPTION A Three summary cards One card per signal with drill-in. Calm, scannable. Defaults well. OPTION B Insights feed Reverse-chronological stream of detected issues. Feels like Jira Insights. Sketch in A's design notes. OPTION C Tree heat-map Taxonomy tree colored by health. Beautiful but information-dense. Sketch in A's design notes.
What we need feedback on: all three signals route to the same Suggestions Inbox. Should Tag Health be its own dashboard, or just live as a "filter by Reason" on the inbox?

About this deck

Each mockup is standalone HTML+CSS+vanilla JS. Real-looking fake data. Shared mockup.css drives consistency. Mermaid diagrams only on this index. Tabs and chips have working hover states; reviewers can click between options and tabs without anything breaking. Honesty note: the deck delivers 12 polished pages of an aspirational 18 — three options each for Suggestions Inbox, Source Configuration and Taxonomy Admin (the highest-traffic surfaces), and one polished representative (Option A) for Tag Scope, Synonyms and Tag Health. The B/C variants for those three surfaces are described in the index callouts and in each A page's "Design notes" section, but not yet built — the picked direction can drive that work. Deferred options are visually marked with a hatched pattern and "NOT BUILT" label so reviewers don't click into broken links.
