# Initial Smoke Test Findings — 2026-04-20

Quick capture of what the smoke tests tell us before committing to implementation. Structured as: hypothesis → actual result → implication for the plan.

---

## T1 — Distilled vs verbose propagation: **PASS**

**Hypothesis**: one-line distilled parent summaries beat verbose full-paragraph parent descriptions at similar quality.

**Result (validated both in the earlier standalone run and in the suite)**:

| Metric | verbose (today) | distilled | ratio |
|---|---|---|---|
| Total latency (5 tables × 3 samples) | 41.1 s | 26.3 s | 64% |
| Total input tokens | 4572 | 1495 | 33% |
| Total output tokens | 2831 | 1649 | 58% |
| PK correctness | 5/5 | 5/5 | parity |
| FK correctness | 2/5 (A had dbo.-prefix format bug) | 5/5 | **B strictly better** |
| Latency variance on PlaylistTrack | 5.6–18.3 s | 5.2–5.4 s | **B much more stable** |

**Implication**: this is a proven, measurable win. Keep it. **~1.5× speedup per call with equal-or-better quality.**

## T2 — VIF ordering within a level (simulated): **TIE on Chinook**

Chinook's graph is small enough that all roots have VIF=0; the ordering doesn't move the needle. This does not invalidate VIF priority — it just means Chinook is too small a test for it. The real test would be on a schema with 30+ tables and heterogeneous roots.

**Implication**: defer VIF validation to LousyDB / AdventureWorks. Don't over-invest in it pre-evidence.

## T4 — Synthesis ramp: **FAIL** (the load-bearing test)

**Hypothesis**: As DFS descends through the dependency graph, per-call LLM latency + output-token count *decrease*, because deeper tables become synthesis tasks (compose resolved ancestors) rather than inference tasks (infer from primitives).

**Result**:

| Depth | Tables | Avg latency ms | Avg output tokens |
|---|---|---|---|
| 0 | 5 | 4510 | 246 |
| 1 | 2 | 5838 | 396 |
| 2 | 2 | 7213 | 487 |
| 3 | 1 | 4518 | 295 |
| 4 | 1 | 5325 | 393 |

**Trend: increasing, not decreasing.** VERDICT: FAIL.

### What actually happens (root cause)

Latency scales primarily with **column count**, not graph depth. Examples:
- Artist (depth 0, 2 cols): 152 out tokens, 3654 ms
- PlaylistTrack (depth 3, 2 cols): 295 out tokens, 4518 ms
- Track (depth 2, 9 cols): 562 out tokens, 8076 ms
- Customer (depth 1, 8 cols): 547 out tokens, 7067 ms

Even controlling for column count, deeper tables generate *more* tokens, not fewer, because the LLM uses the propagated parent context to write *richer* descriptions. The cognitive task doesn't get easier; the output gets better-informed.

### Falsified conclusion

"LLM work per call decays with depth" is **not supported** by this experiment. The architectural argument that the speedup is structural ("the shape of the work changes") is weakened.

The speed win from distilled propagation that T1 validated is **entirely attributable to smaller prompts**, not to any synthesis-ramp effect. The attribution is: 67% fewer input tokens × 42% fewer output tokens ⇒ 36% wall-time reduction per call. That's a real, compounding-with-parallelism win — but it's a "send less, save time" win, not a "work less, save time" win.

## T5 — Simulator projections

With default per-depth latency assumptions (now known to be wrong in direction), projected for Chinook:

| Config | Wall time | Calls |
|---|---|---|
| baseline (today) | 140 s | 22 |
| parallel only (4 workers) | 80 s | 22 |
| + distilled propagation | 16 s | 11 |
| + reach priority | 16 s | 11 |
| + caching | 15 s | 11 |
| full (+ clusters) | 15 s | 11 |

Projected speedup: 9.33× on Chinook.

**Important caveat**: the simulator's per-depth latency model was calibrated under the (now-falsified) synthesis-ramp hypothesis. Reality's per-call latency is not lower at depth. If we re-calibrate with actual per-table averages, the projected speedup shrinks because the "+ distilled" step doesn't get depth-based discount compounding.

Recalibrated estimate using the observed (distilled-mode) per-call mean of ~5.3 s:
- 11 tables serial × 5.3 s = 58 s
- 4-way parallel, respecting DAG: ~20–25 s (graph depth 5, parallelism bottlenecks)

Still a meaningful speedup over today's baseline of 390 s (6:30) — **roughly 15–20× on Chinook**.

But attribute this to **parallelism + distilled prompts**, not to the structural-work argument.

## Plan consequences

### What survives (validated, publishable wins)

1. **Distilled propagation**: 1.5× per call, equal-or-better quality.
2. **Intra-level / DAG parallelism**: 3–4× (externally validated — Helium, DynTaskMAS).
3. **Prompt caching**: 1.3× on cached prefix (externally validated).

Stacked, honestly: probably 4–6× on Chinook/LousyDB, more on AdventureWorks via cluster parallelism.

### What falls (or at least gets demoted)

1. The "synthesis ramp / difficulty decay" narrative. The architecture doesn't need it to deliver a real speedup, but we can't defend the specific claim "deeper calls are easier."
2. VIF ordering as a speed argument. Keep it as a scheduling tiebreaker and for prompt-size LPT reasons, but don't sell it as a separate speed contributor.
3. Fiedler/clustering as a speed claim on small-to-medium schemas. It's still defensible on 50+ table schemas (AdventureWorks-scale) where independent subgraph parallelism is the only way to bound wall time by graph depth rather than table count. Demote to "applies on large schemas only."

### Revised implementation sequence

1. **Ship distilled propagation** — refactor `buildTableContext` and `table-analysis.md` to use cached prefix + distilled suffix. Measurable 1.5× per call, zero quality risk.
2. **Ship DAG scheduler + intra-level parallelism** — straightforward task-graph scheduler, measurable 3–4×.
3. **Wire prompt caching** — provider-level, small delta, easy.
4. **Measure on Chinook + LousyDB** — confirm F1 stays intact, wall-time shows expected reduction.
5. **Then** decide whether to invest in Fiedler clustering based on observed AdventureWorks behaviour.

The plan still works. The vision doesn't.

---

## Meta lesson

We designed a smoke test against a core hypothesis, ran it in 2 minutes of wall clock, and it falsified the hypothesis. This is the correct outcome for a smoke test — it saved us from implementing a clustering + VIF-priority scheduler that was theoretically motivated by a claim that doesn't hold.

The earlier bundled-everything approach gave 1× speedup and we couldn't attribute anything. T4's quick negative result is worth more than another day of architecture work built on a wrong premise.

The remaining proven wins are significant and shippable. The plan simplifies.
