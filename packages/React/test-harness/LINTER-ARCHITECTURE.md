# Component Linter Architecture Guide

This document describes the architecture of the MemberJunction interactive component linter after the Phase 4 modularization refactoring. It is intended for developers who need to understand, maintain, or extend the linter.

## Overview

The component linter validates React interactive components generated by the Skip-Brain AI system. These components run inside a sandboxed runtime where they cannot import modules, export values, or access `window` — all dependencies are injected as props (`utilities`, `styles`, `components`, `callbacks`). The linter enforces these constraints and catches common mistakes before components are executed.

### Key numbers
- **62 lint rules** in individual self-registering files
- **~1,400 line orchestrator** (`component-linter.ts`)
- **303 test fixtures** (118 broken, 96 fixed, 37 valid, 48 future, 4 missing refs)

## Core Concepts

### LintRule Interface

Every rule implements `LintRule` (defined in `lint-rule.ts`):

```typescript
interface LintRule {
  name: string;                    // Unique rule identifier, e.g. 'no-import-statements'
  appliesTo: 'all' | 'child' | 'root';  // Which components this rule applies to
  deprecated?: boolean;            // Mark rules being phased out
  test: (
    ast: t.File,                   // Parsed Babel AST
    componentName: string,         // Name of the component
    componentSpec?: ComponentSpec,  // Component metadata (queries, events, deps)
    options?: ComponentExecutionOptions
  ) => Violation[];                // Array of violations found
}
```

### RuleRegistry

The `RuleRegistry` singleton manages all rules. Rules self-register when their module is imported:

```typescript
// In each rule file:
RuleRegistry.getInstance().registerRuntimeRule(myRule);

// In the orchestrator (component-linter.ts):
import './runtime-rules';  // Side-effect import triggers all registrations
const rules = RuleRegistry.getInstance().getRuntimeRules();
```

The registry prevents duplicate registration (by rule name).

### Violation

Every rule produces `Violation` objects:

```typescript
interface Violation {
  rule: string;           // Rule name that produced this violation
  severity: 'critical' | 'high' | 'medium' | 'low';
  line: number;
  column: number;
  message: string;
  code?: string;          // Code snippet that caused the violation
  suggestion?: {
    text: string;         // Human-readable fix description
    example?: string;     // Code example showing correct pattern
  };
}
```

## How to Add a New Rule

### 1. Create the rule file

Create `runtime-rules/my-new-rule.ts`:

```typescript
import traverse, { NodePath } from '@babel/traverse';
import * as t from '@babel/types';
import { LintRule } from '../lint-rule';
import { RuleRegistry } from '../rule-registry';
import { Violation } from '../component-linter';
import { createViolation } from '../lint-utils';

/**
 * Rule: my-new-rule
 *
 * Description of what this rule checks.
 *
 * Severity: critical | high | medium | low
 * Applies to: all | root | child
 */
export const myNewRule: LintRule = {
  name: 'my-new-rule',
  appliesTo: 'all',
  test: (ast, componentName, componentSpec, options) => {
    const violations: Violation[] = [];

    traverse(ast, {
      // Visit AST nodes and push violations
      CallExpression(path: NodePath<t.CallExpression>) {
        // Your validation logic here
        violations.push(
          createViolation(
            'my-new-rule',
            'high',
            path.node,
            `Description of what went wrong in "${componentName}"`,
            path.toString(),
            'How to fix it'
          )
        );
      },
    });

    return violations;
  },
};

// Self-register when this module is imported
RuleRegistry.getInstance().registerRuntimeRule(myNewRule);
```

### 2. Add the import to index.ts

Add one line to `runtime-rules/index.ts`:

```typescript
import './my-new-rule';
```

That's it. The rule will automatically be picked up by the orchestrator on the next lint run.

### 3. Add test fixtures

Create fixture files to test the rule:

- **Broken fixture** (`fixtures/component-linter/broken-components/`): A component that should trigger the rule. The test asserts at least one violation is produced.
- **Valid fixture** (`fixtures/component-linter/valid-components/`): A component that should NOT trigger the rule. The test asserts zero violations.

Fixture format (JSON):
```json
{
  "name": "my-broken-component",
  "code": "function MyComponent({ utilities, styles }) {\n  // code that violates the rule\n  return <div>test</div>;\n}"
}
```

Or use `$ref` to point to a real component spec:
```json
{
  "$ref": "metadata/components/spec/some-component.spec.json",
  "description": "Tests my-new-rule against a real component"
}
```

### 4. Run tests

```bash
cd packages/React/test-harness
npx vitest run
```

## Rule Severity Guidelines

| Severity | When to use | Example |
|----------|-------------|---------|
| `critical` | Breaks the runtime sandbox or causes crashes | Import statements, export statements, IIFE wrappers |
| `high` | Likely runtime error or incorrect behavior | Wrong RunView property names, invalid entity fields |
| `medium` | Code quality issue, potential bug | Case mismatches, unnecessary patterns |
| `low` | Style preference, best practice | Prefer async/await over .then() |

## Rule Scope (`appliesTo`)

- **`all`**: Rule applies to both root and child components. Most rules use this.
- **`root`**: Only the top-level component (the one matching the spec name). Used for rules like `required-queries-not-called` that only make sense for root components.
- **`child`**: Only embedded child components. Used for rules like `no-child-implementation` that prevent child components from implementing certain patterns.

## Existing Rule Categories

### Runtime Constraint Rules
Rules that enforce the sandboxed runtime environment:
- `no-import-statements`, `no-export-statements`, `no-require-statements`
- `no-iife-wrapper`, `no-return-component`, `no-window-access`
- `use-function-declaration`, `single-function-only`

### Component Structure Rules
Rules about how components are structured:
- `react-component-naming`, `component-name-mismatch`
- `pass-standard-props`, `no-react-destructuring`
- `no-data-prop`, `no-child-implementation`

### Data Access Rules
Rules for RunView/RunQuery API usage:
- `runview-runquery-valid-properties`, `runview-entity-validation`
- `runquery-parameters-validation`, `runquery-missing-categorypath`
- `entity-field-validation`, `query-field-validation`
- `runquery-runview-spread-operator`, `runview-runquery-result-validation`

### Component Dependency Rules
Rules about embedded components and libraries:
- `component-not-in-dependencies`, `undefined-component-usage`
- `unused-libraries`, `unused-component-dependencies`
- `library-variable-names`, `dependency-shadowing`

### Callback & Event Rules
Rules for event handling patterns:
- `callbacks-usage-validation`, `callbacks-passthrough-only`
- `callback-parameter-validation`, `event-invocation-pattern`

### Best Practice Rules
Code quality and pattern rules:
- `prefer-async-await`, `prefer-jsx-syntax`
- `react-hooks-rules`, `useeffect-unstable-dependencies`
- `unsafe-array-operations`, `unsafe-formatting-methods`
- `styles-invalid-path`, `styles-unsafe-access`

## Specialized Validation Systems

Beyond the 62 self-registering rules, the linter has three specialized validation systems that run as part of the orchestration pipeline:

### TypeCompatibilityRule (Phase 1)
Runs type inference once via `TypeInferenceEngine` and checks for type mismatches, coercion errors, and incompatible operations across the entire component. This replaced multiple per-rule type inference calls.

### ComponentPropRule (Phase 3)
Validates component props against their spec definitions. Checks prop names, types, required props, and constraint validators. Uses `SemanticValidatorRegistry` for constraint-based validation (SQL WHERE clauses, entity field subsets, conditional requirements).

### Library Lint Rules
Database-driven rules loaded from `MJComponentLibrary.LintRules`. These validate library-specific patterns (initialization, lifecycle, ref types, options). Loaded via `LibraryLintCache` and require a `contextUser` for database access.

## Test Fixtures

Fixtures live in `fixtures/component-linter/`:

| Directory | Purpose | Test assertion |
|-----------|---------|----------------|
| `broken-components/` | Components with known violations | Must produce ≥1 violation |
| `fixed-components/` | Components that were fixed | Must NOT produce invalid-property violations |
| `valid-components/` | Correct components | Must produce 0 violations |
| `future-components/` | Tests for unimplemented rules | Not executed (reserved for future work) |

### Running tests with database

Some fixtures require a database connection for entity/query metadata validation. Create a `.env` file in `packages/React/test-harness/`:

```
DB_HOST=your-host
DB_DATABASE=your-db
DB_USERNAME=your-user
DB_PASSWORD=your-password
DB_PORT=1433
DB_TRUST_SERVER_CERTIFICATE=1
```

The vitest setup file (`src/__tests__/setup/vitest-setup.ts`) automatically connects and initializes the MJ metadata provider. Without a database, DB-dependent fixtures run but rules that need entity metadata will silently skip validation.

## Future Work

- **Visitor merging**: Currently each rule calls `traverse(ast, {...})` independently (62 AST walks). A future optimization would have rules declare which node types they visit, and the orchestrator would merge them into a single traversal. This is how ESLint works.
- **Future fixture rules**: The 48 fixtures in `future-components/` represent validation rules not yet implemented: cross-component type flow, entity field validation in array operations, chart field checking, query result field validation, and data grid field validation.
- **Phase 5 consolidation**: Seven RunQuery/RunView rules could be merged into 2 combined rules. This is optional polish.
