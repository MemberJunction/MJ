**Branch:** `connectors/integration-v2-unified`
**Audience:** a skeptical PR reviewer. Every architectural claim below cites `file:line` (paths relative to repo root). Every capability is labelled **[proven]** (a test or a live/mock run exercises it), **[implemented-unproven]** (code exists, nothing exercises it), or **[partial]**. No claim is made without a citation.

- `packages/Integration/pk-classifier`: **9 passed**.
- `packages/Integration/schema-builder`: **98 passed / 3 FAILED** — `TypeMapper.test.ts` has 3 stale-drift failures that encode the *old* bounded-string policy. This is a real finding, documented in §4. It is not cosmetic — it is the test-side half of the NVARCHAR(MAX) gap.

Terminology: `__mj` is the MJ core schema. Each connector's synced tables live in a **per-connector SQL schema** (e.g. `growthzone`, `orcid`), isolated from `__mj`. "Soft PK" = a primary key that exists only in MJ metadata + `additionalSchemaInfo.json`, never as a DB `PRIMARY KEY`/`UNIQUE` constraint.

## 1. End-to-end data flow

The lifecycle is: create a connection → discover the source schema → materialize that schema as real SQL tables + MJ entities + CRUD sprocs (`ApplyAll`) → run a sync → fetch/paginate → map → upsert-by-identity → incremental skip (watermark + content-hash) → capture unmapped fields to an overflow column → promote pervasive overflow keys to real columns post-sync.

### 1a. `IntegrationCreateConnection` — establish the connection
GraphQL mutation at `packages/MJServer/src/resolvers/IntegrationDiscoveryResolver.ts:2427` (method `IntegrationCreateConnection`, def `:2428`). It creates a `Credential` row (`:2440-2451`), then a `CompanyIntegration` linked to it (`:2454-2469`), optionally runs a connection test (`:2471-2487`), and optionally auto-runs the discovery pipeline (`:2494-2505`). **[proven]** — the live PropFuel/HubSpot runs created connections through this surface (`packages/Integration/connectors/test/PROPFUEL_LIVE_FINDINGS.md`, "GQL endpoints (11 read + 5 lifecycle mutations)").

### 1b. Discovery — `IntegrationConnectorCreationPipeline`
`IntegrationRefreshConnectorSchema` (`IntegrationDiscoveryResolver.ts:1211`, def `:1212`) instantiates `IntegrationConnectorCreationPipeline` (`:1222`) and runs it (`:1236`). The pipeline drives `IntrospectSchema → PersistDiscoveredSchema → SoftPKClassifier` (documented `:345-348`). Connectors implement `DiscoverObjects`/`DiscoverFields` (abstract on the base, `BaseIntegrationConnector.ts:679` and `:691`). For sources with **no** describe endpoint there is a read-path discovery: `DiscoverFieldsViaStream` (`BaseIntegrationConnector.ts:530`) and `DiscoverFieldsViaFetch` (`:606`) stream real records read-only and infer fields + a provable PK from statistics. **[proven]** — `DiscoverFieldsViaStream.test.ts`, `StreamingDiscovery.test.ts` (in `packages/Integration/engine/src/__tests__/`); discovery materialized 130+ HubSpot objects / 282 contact fields live (`NIGHT_SUMMARY.md` §3).

### 1c. `ApplyAll` — materialize the schema (the heavy stage)
`IntegrationApplyAll` (`IntegrationDiscoveryResolver.ts:3027`, def `:3028`):
1. Reconstructs `SourceSchemaInfo` from the persisted IntegrationObject/Field rows, with a live-introspect fallback (`:3052-3116`).
2. Builds the schema via `SchemaBuilder` (`:3136-3138`, `buildSchemaForConnector` `:3466-3548`).
3. Runs the RSU pipeline: `RuntimeSchemaManager.Instance.RunPipelineBatch([rsuInput])` (`:3172-3173`).
4. Creates entity + field maps (`createEntityAndFieldMaps` `:3281`, called `:3196-3199`).
5. Optionally kicks the first sync (`startSyncAfterApply` `:3206-3210`).

**RSU pipeline internals** — `packages/SchemaEngine/src/RuntimeSchemaManager.ts`. The class is a `BaseSingleton` (`:311`, `Instance` accessor `:316-318`), gated by `ALLOW_RUNTIME_SCHEMA_UPDATE === '1'` (`:447-449`). `RunPipelineBatch` (`:517-541`) runs four phases: validate (`:529`), execute migrations (`:533`), post-migration pipeline (`:537`), build results (`:540`). The post-migration core (`runPostMigrationPipeline` `:609-664`) is the ordered RSU loop: write `additionalSchemaInfo` (`:618`) → **RunCodeGen** (`:620`) → compile TypeScript (`:633`) → git commit/PR (`:645`) → restart MJAPI (`:659`). CodeGen is what generates the per-entity `spCreate/spUpdate/spDelete` sprocs and views and registers the new MJ entities.

DDL is produced by `SchemaBuilder`, which delegates to **SchemaEngine's** `DDLGenerator` (`packages/Integration/schema-builder/src/SchemaBuilder.ts:85-91`, imported as `GenericDDLGenerator` `:18-27`) — *not* schema-builder's own `DDLGenerator.ts` for table creation in that path. Idempotent creation uses `IfNotExists: true` (`SchemaBuilder.ts:91`).

**[partial]** — the RSU + ApplyAll path is **live-proven on SQL Server** end-to-end (`PROPFUEL_LIVE_FINDINGS.md`: full pull of 60,164 records into materialized tables; `NIGHT_SUMMARY.md`). It is **not** proven on a fresh Postgres DB in this loop — see §4 (PG baseline blocker) and the project's own dual-dialect policy in `.claude/rules/connector-test-conventions.md`.

### 1d. `StartSync` → engine `RunSync`
`IntegrationStartSync` (`IntegrationDiscoveryResolver.ts:3556`, def `:3557`) rejects up front if `IsActive=false` (`:3576-3578`), then fire-and-forget calls `IntegrationEngine.Instance.RunSync(...)` (`:3593-3600`). The engine creates the `CompanyIntegrationRun` row, not the resolver; the resolver then **polls** for that row (`:3636-3652`) and returns `Success:false` if none appeared (`:3657-3662`) — this is a deliberate anti-silent-no-op guard. Engine entry point: `IntegrationEngine.RunSync` (`packages/Integration/engine/src/IntegrationEngine.ts:376`), delegating to `executeSyncInternal` (`:419`). **[proven]** for the run-creation + polling path (the `IntegrationEngine.*.test.ts` suite drives `RunSync` to a complete run with counts — see the ratelimit-wiring test output: `sync.run.complete ... success:true`).

> **Caveat (known gap #8):** when a prior promotion flagged `restartRequiredForGraphQL`, `StartSync` can return `Success:true, RunID:null` and start nothing — a misleading success. Documented in `INTEGRATION_KNOWN_GAPS.md` §8; the resolver's own poll-then-fail guard (`:3657-3662`) catches the *no-row* case but the restart-pending case is the open one.

### 1e. Fetch / paginate
`ExecuteEntityMaps` (`IntegrationEngine.ts:786`) orders maps into DAG layers (`buildEntityMapDependencyLayers` `:958`, Kahn) and runs **layers sequentially**, maps **within a layer concurrently** up to an AIMD cap (`:889-944`). The fetch loop (`while (hasMore)` `:1505`) calls `config.connector.FetchChanges(ctx)` (`:1557`) wrapped in retry + timeout (`:1555-1560`). The `FetchContext` carries pagination state including keyset `AfterKeyValue` (`:1529`); state advances per batch (`:1763-1766`), e.g. `currentAfterKey = batch.NextAfterKeyValue ?? currentAfterKey` (`:1766`). **[proven]** — `IntegrationEngine.test.ts`, `WithTimeout.test.ts`, `RetryRunner.test.ts`; live full pull of 60k records.

### 1f. Map + capture unmapped fields
`FieldMappingEngine.Apply` (`IntegrationEngine.ts:1675-1677`) produces `MappedRecord`s carrying both `MappedFields` (shortlisted) and `UnmappedFields`. Unmapped keys are computed by `computeUnmappedFields` (`packages/Integration/engine/src/CustomOverflow.ts:59`) = full source keys − active map source-field names. They are persisted to the overflow column only when non-empty: `IntegrationEngine.ts:3611-3612` writes `entity.Set(CUSTOM_OVERFLOW_COLUMN, JSON.stringify(record.UnmappedFields))` gated by `hasField(CUSTOM_OVERFLOW_COLUMN) && hasUnmappedFields(...)`. Column name: `__mj_integration_CustomOverflow` (`CustomOverflow.ts:41`). Only mapped fields are written to the entity — `SetEntityFields` iterates `fields` (the mapped object) only (`IntegrationEngine.ts:3426-3453`). **[proven]** — `CustomOverflow.test.ts` (8 tests); live "custom-column capture (blacklist→overflow)" (`PROPFUEL_LIVE_FINDINGS.md`).

### 1g. Upsert-by-identity
Matching is by external identity via `MatchEngine.Resolve` (`IntegrationEngine.ts:1700`). Create path (`CreateRecord` `:3059`): extract mapped PK (`:3073`), load-by-PK, and only `NewRecord()` if absent (`:3074-3076`) — idempotent upsert. Update path (`UpdateRecord` `:3172`). Identity itself: `CompanyIntegrationRecordMap` rows are 1:1 per `(CompanyIntegration, Entity, ExternalSystemRecordID)`. **[proven]** — `MatchEngine.test.ts`, `IntegrationEngine.recordmap-update.test.ts`; live: re-fetch of ~20k records → **0 created**, `total == distinct` across all objects (the duplicate bug fixed; `PROPFUEL_LIVE_FINDINGS.md`).

### 1h. Incremental — watermark (primary) + content-hash (fallback)
Watermark is the **primary** mechanism. `WatermarkService.Load` (`IntegrationEngine.ts:1388`) reads the cursor; `ValidateWatermark` (`:1408`) rejects format mismatch; the max-seen watermark is persisted on full success only (`watermarkService.Update` `:1865`), and the safe floor advances only when a batch applied with zero new errors (`:1761`) so a rolled-back batch stays re-fetchable. Content-hash is the **fallback** for sources with no usable watermark: `computeContentHash(record.MappedFields)` (`ContentHash.ts:30`) is compared against the stored `__mj_integration_ContentHash` at the create-upsert precheck (`IntegrationEngine.ts:3092`), the update precheck (`:3197`), and persisted (`:3604`); a batch prefetch (`PrefetchContentHashes` `:3280`) turns N per-record loads into one query. When no hash column exists, the **dirty-flag** path applies — `SetEntityFields` then skip if `!entity.Dirty` (`:3104-3111`, `:3224-3250`). **[proven]** — `WatermarkService.test.ts` (39), `ContentHash.test.ts` (10), `IntegrationEngine.safefloor.test.ts` (3); live: 19,768 records content-hash-skipped on a re-fetch, watermark save-on-cancel + resume.

> Caveat: live run found incremental **narrowing** re-fetched ~20k records on one object because the watermark wasn't saved at the object max in one scenario (`PROPFUEL_LIVE_FINDINGS.md`, "Incremental narrowing — CONFIRMED real gap"). The skip-on-write (content-hash) still made it correct (0 created), but it re-fetched more than ideal. **[partial]** on fetch-narrowing efficiency.

### 1i. Post-sync custom-column promotion
After a sync drains, the engine invokes a registered hook (no engine→RSU dependency): `CustomColumnPromoter` (`packages/MJServer/src/integration/CustomColumnPromoter.ts:1-23`). It (1) gates on overflow rows existing (`:8-9`), (2) plans which keys earn a column with a generously-bounded type (`buildOverflowStats`/`planPromotions`, capped at `MAX_PROMOTIONS_PER_PASS=25` `:64`), (3) ADD COLUMNs via `DDLGenerator` + `RuntimeSchemaManager` (which re-runs CodeGen to regenerate `spUpdate<Entity>` with the new params), writes an IOF (`IsCustom`, `MetadataSource='Discovered'`), and a field map so the next sync maps it natively and the loop converges. **[proven]** at unit level — `CustomColumnPromotion.test.ts` (23 tests, the plan/stats/sanitize logic); the ADD COLUMN + sproc-regeneration end-to-end is **[partial]** (proven on a fresh dist-booted MJAPI; flaky on a long-lived in-process-CodeGen MJAPI — `INTEGRATION_KNOWN_GAPS.md` §6/§9).

## 2. How it adheres to MemberJunction conventions

- **BaseEngine / cached metadata.** `IntegrationEngineBase extends BaseEngine` (`packages/Integration/engine-base/src/IntegrationEngineBase.ts:25`), `@RegisterForStartup()` (`:24`), `BaseSingleton` via `Instance` (`:342-344`). It caches all integration entities with `CacheLocal: true` (`:38-83`) — the canonical MJ engine pattern. **[proven]** by usage in the test suite.
- **CodeGen-generated entities, no hand-rolled CRUD.** RSU runs CodeGen (`RuntimeSchemaManager.ts:620`) so synced tables get the standard generated `spCreate/spUpdate/spDelete`, views, and registered entity classes — the framework never hand-writes CRUD SQL for synced tables. **[proven]** (live tables materialized + populated).
- **`RegisterClass` factory dispatch.** Connectors register via `@RegisterClass(BaseIntegrationConnector, 'DriverClass')` (e.g. `packages/Integration/connectors/src/ORCIDConnector.ts:143`). `ConnectorFactory` resolves through `MJGlobal.Instance.ClassFactory.GetRegistration(BaseIntegrationConnector, driverClass)` (`packages/Integration/engine/src/ConnectorFactory.ts:41-54`). The base classes themselves are *not* decorated — registration is grandparent-keyed on concrete connectors, per `.claude/rules/connector-code-conventions.md`. **[proven]** (factory is exercised by every connector test).
- **Per-connector schema isolation.** Synced tables live in a dedicated SQL schema per connector, separate from `__mj`; access to `__mj` writes is gated (`FRAMEWORK_TESTABLE_CAPABILITIES.md` "Access gate"; `AccessControl.ts` in schema-builder). **[implemented-unproven]** at the access-gate-error level here (covered by `AccessControl.test.ts` but I did not re-run that assertion to ground it as live).
- **Soft PK via `additionalSchemaInfo`.** Integration tables carry **no** DB `PRIMARY KEY`/`UNIQUE` constraint on inferred keys (`packages/Integration/schema-builder/src/DDLGenerator.ts:28-30`, `:61-62`: "enforcing one — even as UNIQUE — would reject valid rows the moment an inference is wrong"). The PK lives in `additionalSchemaInfo.json`, which CodeGen reads to set the entity's metadata PK + `IsSoftPrimaryKey` flag (`:28-30`). RSU writes that file (`RuntimeSchemaManager.ts:360-381`). **[proven]** — `SoftFKConfigEmitter.test.ts`, `DDLGenerator.test.ts` (emits non-unique index, never a constraint); live tables had no rejected rows.
- **UUID dialect-safety.** Lookups use `UUIDsEqual` throughout `IntegrationEngineBase` (`:174, 185, 190, 195, ...`), per the MJ UUID-comparison rule.
- **Full-record pass-through (forward-compat).** `ToExternalRecord` sets `Fields: raw` (the full source record), `BaseRESTIntegrationConnector.ts:1163-1170`; the framework persists only `MappedFields` so a full `Fields` never causes unknown-column writes (`INTEGRATION_KNOWN_GAPS.md` §0). **[proven]** by base path; **[partial]** for the 23 override-fetch connectors (22/23 pass full record; `YourMembershipConnector` is the documented exception, gaps §0).

## 4. Known limitations / honest gaps

These are real. Several are documented by the team in `INTEGRATION_KNOWN_GAPS.md`; I reproduce them accurately and add what I independently found.

1. **Bounded-typing is partial — unsized strings become `NVARCHAR(MAX)`, and the unit tests are stale-drifted (FOUND THIS RUN).**
   `TypeMapper.MapSourceType` deliberately emits the dialect's unbounded text type for strings with no declared length (`TypeMapper.ts:67-69`), with a documented rationale (avoid truncation + avoid `SchemaEvolution` trying to narrow an existing MAX column). The consequence: APIs that don't report field sizes (GrowthZone, PropFuel) get non-PK `NVARCHAR(MAX)` columns. Independently corroborated by the **live SQL Server run** (`PROPFUEL_LIVE_FINDINGS.md`: "non-PK columns are `NVARCHAR(MAX)`"). **Additionally, `schema-builder` has 3 FAILING unit tests** (`TypeMapper.test.ts`: "should default string without MaxLength to the 255 floor (never MAX)", the PK-cap test expecting 450 vs received 750, and a case-insensitive variant). The code was changed; the tests were not — a violation of the repo's CRITICAL RULE #6 (run + update unit tests). Only the **PK** column is reliably bounded (capped to 450, `DDLGenerator.ts:48-51`), so dedup is unaffected; non-PK MAX is wasteful, not incorrect. **Net: the "no blanket NVARCHAR(MAX)" claim is not fully true today, and the test suite for this module is red.**

2. **In-process CodeGen schema-cache staleness (gaps #6 / #9).** A long-lived MJAPI using the in-process CodeGen runner caches `additionalSchemaInfo.json` / schema state across runs, so a later connector's soft-PKs or a just-ALTERed table can be invisible to CodeGen — symptoms: "No primary key found" for tables whose PK *was* written, or an `spUpdate` sproc missing just-minted columns. Workaround: a **fresh / dist-booted** process regenerates correctly (proven on ORCID dist-boot); production ApplyAll uses the **child-process** runner (fresh each call). `RuntimeSchemaManager.ts:1062-1118` chooses in-process (`:1064`) vs child-process (`:1073-1074`). **[partial]** — the designed flow works on a fresh process; the in-process reuse is a live-observed defect, not yet fixed.

3. **Dev-mode (tsx) MJAPI cannot load freshly-RSU-generated resolvers (gap #7).** tsx/esbuild doesn't support parameter decorators + `emitDecoratorMetadata`, so the canonical post-ApplyAll boot must be from **DIST** (`turbo build --filter=mj_api` + `node dist/index.js`). Operational, not a code defect, but it gates how the framework is run.

4. **`StartSync` silent no-op while restart pending (gap #8).** Already noted in §1d — returns `Success:true, RunID:null` and starts nothing when `restartRequiredForGraphQL` is set. Violates the framework's own no-silent-failure rule; fix belongs in the StartSync gate.

5. **Postgres is not live-proven in this loop.** Per `.claude/rules/connector-test-conventions.md`, PG is **suspended** for the per-connector loop because a fresh PG install can't CodeGen (the v5.34/v5.37 PG baselines lack the CodeGen sprocs and the fix migrations sit below the baseline). DDL/dialect logic is unit-proven on PG shapes; **no live PG sync ran**. SQL Server is the live axis. This is a meaningful gap for anyone expecting "works on Postgres" out of the box.

6. **Merkle / partition hash-diff is BUILT but opt-in, not the default (gap #3).** `HashDiff.ts` is complete and unit-tested (`HashDiff.test.ts`, 20 tests) but only fires when `Configuration.partitionReconcile` is set. The per-record content-hash skip (the common case) *is* wired. **[implemented-unproven]** as a live default.

7. **Cross-layer pipelined concurrency NOT implemented (gap #4).** Maps parallelize within a DAG layer; layers are strictly sequential.

8. **Push / bidirectional / 3-way merge / conflict resolution is the highest-risk surface and is largely unproven live.** `FRAMEWORK_TESTABLE_CAPABILITIES.md` lists these as `[NEW]` and explicitly flags them as "ONLY provable with writes." The live runs documented here were **pull-only** (PropFuel, ORCID, HubSpot read). Conflict-recency logic has unit coverage (`ConflictRecency.test.ts`), but the end-to-end write-back + conflict path has no live evidence in this branch's docs. **[implemented-unproven]** end-to-end.

9. **`PostProcessRecord` is the wrong home for dialect-dependent coercion (team-flagged).** `BaseIntegrationConnector.ts:494` hook exists, but the team's own review notes destination coercion should live in the SQL-dialect write layer, not a connector hook (`FRAMEWORK_TESTABLE_CAPABILITIES.md` discussion notes). Flagged for cleanup.

## Bottom line for the reviewer

The **read/pull pipeline on SQL Server is genuinely production-grade and well-tested**: discovery, schema materialization via RSU+CodeGen, upsert-by-identity, watermark + content-hash incremental, soft-PK firewall, adaptive rate-limiting, and the custom-column capture/promote loop are all unit-proven (458 engine tests green) and most are corroborated by a 60k-record live run with zero dupes and zero errors. The **weak spots are honest and specific**: bounded typing regresses to `NVARCHAR(MAX)` for unsized strings (with a red test suite proving the drift), Postgres has no live proof in this loop, the write/bidirectional/merge surface is implemented but unproven live, and three in-process-CodeGen / dev-boot operational gaps require a dist boot or process restart. None of these are hidden — they're tracked in `INTEGRATION_KNOWN_GAPS.md` — but they mean "production-ready" applies confidently to **pull-on-SQL-Server**, and with caveats elsewhere.
