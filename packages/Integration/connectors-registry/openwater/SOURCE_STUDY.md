### Authentication Pattern

All endpoints require two required HTTP headers:
- `X-ClientKey` (required) — identifies the client application
- `X-ApiKey` (required) — authenticates the user/service account
- `X-OrganizationCode` (optional) — multi-org scoping
- `X-SuppressEmails` (optional) — side-effect control

No OAuth flow. Header-key scheme throughout.

### Pagination Pattern

All collection GETs use offset pagination via `pageIndex` (optional, default 0) and `pageSize` (optional) query params. All return a `Pagination.PagingResponseXxxListItemModel` schema. `PagingInfo` schema has fields: `pageIndex`, `pageSize`, `totalCount`, `totalPageCount`.

### Form Template Pattern

The spec exposes read-only template GETs (singleton responses, not paginated) that return form schemas:
- `GET /v2/Programs/{programId}/SessionFormTemplate`
- `GET /v2/Programs/{programId}/JudgeProfileFormTemplate`
- `GET /v2/Rounds/{roundId}/SubmissionFormTemplate`
- `GET /v2/Rounds/{roundId}/EvaluationFormTemplate`
- `GET /v2/Users/ProfileFormTemplate`

These return `Models.Program.FormTemplateResponse` / `Models.Round.*FormTemplateResponse` containing field definitions (`Models.Fields.*`). They are INFORMATIONAL — they describe the dynamic field schema of a Program/Round, not a syncable record set.

### Naming Conventions

- PagingResponse wrappers: `Pagination.PagingResponse{Resource}ListItemModel`
- List item models: `Models.{Resource}.{Resource}ListItemModel`
- Detail models: `Models.{Resource}.DetailsResponse`
- Request models: `Models.{Resource}.CreateRequest`, `Models.{Resource}.UpdateRequest`

### Idiosyncrasies

1. **Rounds are embedded, not standalone.** `Models.Program.ProgramListItemModel` has a `rounds` array field (type `Models.Program.RoundModel[]`). There is no `/v2/Rounds` list GET — rounds data is only available by reading the Programs collection and extracting the embedded rounds array. This makes Rounds a NESTED leaf reached via the Programs DOOR.

2. **JudgeAssignments use required query params, not path params, for parent scoping.** `GET /v2/JudgeAssignments/AssignedToRound` requires `roundId` as a query param (not in the path), yet it is NESTED — the collection is always scoped to a round. This is an unusual pattern vs. REST convention.

3. **Three JudgeAssignment GETs all return `JudgeListItemModel`.** AssignedToRound, AssignedToJudgeTeam, and AssignedToApplication+Round all return the same schema. They represent the same entity (a judge assignment record) queryable via different relationship axes. All three are merged under one `JudgeAssignment` taxonomy leaf with multiple access paths.

4. **Report endpoints are triple-keyed.** `GET /v2/Rounds/{roundId}/ApplicationReports`, `GET /v2/Rounds/{roundId}/JudgeReports`, and `GET /v2/Programs/{programId}/SessionReports` all return `ReportListItemModel`. Report listing is always parent-scoped — merged under one `Report` leaf.

5. **Invoice sub-resources as top-level doors.** `BillingLineItems`, `Payments`, and `Refunds` have their own top-level paginated GETs (`/v2/Invoices/BillingLineItems`, `/v2/Invoices/Payments`, `/v2/Invoices/Refunds`) with no required parent scoping param — they are independent DOORS, not NESTED under Invoice.

6. **DeletedData endpoints.** `Applications/DeletedData` and `Sessions/DeletedData` are separate paginated list GETs returning tombstone list models. Treated as independent DOOR leaves (`DeletedApplication`, `DeletedSession`).

7. **Export endpoints (all POST, not GETs).** `/v2/Export/*` paths are all POST-only trigger/export actions, not syncable record sets. Excluded entirely from COVERABLE taxonomy.

8. **Media: no collection GET.** Only `POST /v2/Media` (upload) and `GET /v2/Media/{id}` (single-item). Excluded.

9. **SessionChairs: zero GET.** Only `POST`, `PATCH`, `DELETE`. Excluded.

## Gaps

No authoritative source covers the following aspects:

| Area | Reason |
|---|---|
| Webhook / event-push API | The spec documents no webhook registration or event subscription endpoints. OpenWater may have outbound webhooks at the platform level but they are not in the v2 REST spec. |
| OAuth / SSO integration details | The `POST /v2/Users/{userId}/SsoToken` endpoint generates an SSO token, but the OAuth flow / token lifecycle is not documented in the spec itself. Help center articles likely cover this but are not machine-readable. |
| Rate limit specifications | No rate limit headers or limit values are documented in the spec. The `6349444-media-data-storage-api-calls-and-other-limits` help article exists but was not confirmed to cover rate limit numeric values for the REST API. |
| Batch write endpoints | No batch create/update endpoints exist in the spec — all writes are per-record. This is an architectural characteristic, not a gap in documentation. |
| Detailed Application.DetailsResponse fields | The list endpoint returns summary fields; full details (roundSubmissions, collaborators, form field values, scheduling info) require a per-ID GET. The sync strategy for detail field capture needs a two-pass approach (list -> detail GET per ID). |
