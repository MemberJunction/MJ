# Schema Management

How the integration framework generates database schemas for integration target tables.

## Overview

When mapping an external object to a **new** MJ entity (one that doesn't exist yet), the schema builder generates:

1. **Migration SQL** — CREATE TABLE / ALTER TABLE statements (Flyway-compatible)
2. **Soft FK configuration** — `additionalSchemaInfo.json` entries for CodeGen
3. **Entity settings metadata** — mj-sync JSON for `IntegrationWriteAllowed` flags

The schema builder **only emits files** — it never executes SQL. A human reviews the generated artifacts, applies the migration, and runs CodeGen.

## Pipeline

```
Source Schema Introspection
        │
        ▼
┌──────────────────┐     ┌─────────────────────┐
│   TypeMapper     │────▶│   DDLGenerator       │──→ CREATE TABLE SQL
│ generic→SQL type │     │ platform-aware DDL   │
└──────────────────┘     └─────────────────────┘
        │                         │
        ▼                         ▼
┌──────────────────┐     ┌─────────────────────┐
│ SchemaEvolution  │────▶│ MigrationFileWriter  │──→ Migration files
│ diff detection   │     │ Flyway naming        │
└──────────────────┘     └─────────────────────┘
                                  │
                                  ▼
                         ┌─────────────────────┐
                         │ SoftFKConfigEmitter  │──→ additionalSchemaInfo.json
                         └─────────────────────┘
                                  │
                                  ▼
                         ┌─────────────────────┐
                         │  MetadataEmitter     │──→ mj-sync entity settings
                         └─────────────────────┘
                                  │
                                  ▼
                         ┌─────────────────────┐
                         │   AccessControl      │──→ Security validation
                         └─────────────────────┘
```

## Type Mapping

The `TypeMapper` converts generic source types to platform-specific SQL types:

| Generic Type | SQL Server | PostgreSQL |
|-------------|------------|------------|
| `string` | `NVARCHAR(255)` | `VARCHAR(255)` |
| `string` (long) | `NVARCHAR(MAX)` | `TEXT` |
| `integer` | `INT` | `INTEGER` |
| `biginteger` | `BIGINT` | `BIGINT` |
| `decimal` | `DECIMAL(18,4)` | `NUMERIC(18,4)` |
| `boolean` | `BIT` | `BOOLEAN` |
| `datetime` | `DATETIMEOFFSET` | `TIMESTAMPTZ` |
| `date` | `DATE` | `DATE` |
| `uuid` | `UNIQUEIDENTIFIER` | `UUID` |
| `json` | `NVARCHAR(MAX)` | `JSONB` |

Length, precision, and scale from source metadata are preserved when available.

## DDL Generation

### CREATE TABLE

Every integration table gets these standard columns automatically:

| Column | Type | Purpose |
|--------|------|---------|
| `ID` | UNIQUEIDENTIFIER / UUID | MJ primary key |
| `SourceRecordID` | NVARCHAR(500) | External system record ID |
| `SourceObject` | NVARCHAR(255) | External object name |
| `SyncedAt` | DATETIMEOFFSET / TIMESTAMPTZ | Last sync timestamp |

Plus all columns from the source schema, type-mapped to the target platform.

**Note**: `__mj_CreatedAt` and `__mj_UpdatedAt` are NOT included — CodeGen adds these automatically.

### What Migrations Do NOT Contain

| Excluded Item | Reason |
|---------------|--------|
| `__mj_CreatedAt` / `__mj_UpdatedAt` columns | CodeGen adds with triggers |
| FK indexes | CodeGen creates `IDX_AUTO_MJ_FKEY_*` |
| Foreign key constraints | Soft FKs via `additionalSchemaInfo` |
| Entity/EntityField INSERT statements | CodeGen discovers from `INFORMATION_SCHEMA` |
| Views (`vw*`) | CodeGen generates |
| Stored procedures (`sp*`) | CodeGen generates |

### Example Output

```sql
-- Auto-generated by MemberJunction Integration Schema Builder v5.7.0
-- Source: YourMembership / Members
-- Platform: sqlserver

IF NOT EXISTS (SELECT * FROM sys.schemas WHERE name = 'ym')
    EXEC('CREATE SCHEMA [ym]');

CREATE TABLE [ym].[Members] (
    [ID] UNIQUEIDENTIFIER NOT NULL DEFAULT NEWSEQUENTIALID(),
    [SourceRecordID] NVARCHAR(500) NOT NULL,
    [SourceObject] NVARCHAR(255) NOT NULL DEFAULT 'Members',
    [SyncedAt] DATETIMEOFFSET NOT NULL DEFAULT GETUTCDATE(),
    [MemberID] NVARCHAR(100) NOT NULL,
    [FirstName] NVARCHAR(255) NULL,
    [LastName] NVARCHAR(255) NULL,
    [Email] NVARCHAR(500) NULL,
    [MemberType] NVARCHAR(100) NULL,
    [JoinDate] DATETIMEOFFSET NULL,
    [Status] NVARCHAR(50) NULL,
    CONSTRAINT [PK_ym_Members] PRIMARY KEY ([ID])
);

CREATE INDEX [IX_ym_Members_SourceRecordID] ON [ym].[Members]([SourceRecordID]);
```

## Schema Evolution

When a source schema changes (fields added, types changed, fields removed), the `SchemaEvolution` class detects differences and generates new migration artifacts.

### Diff Detection

```typescript
const diff = SchemaEvolution.DiffSchema(sourceObj, existingConfig, existingTable, platform);
```

Returns a `SchemaDiff`:

```typescript
interface SchemaDiff {
    AddedColumns: TargetColumnConfig[];
    ModifiedColumns: ColumnModification[];  // type or nullability changes
    RemovedColumns: string[];
}
```

### Field Addition

```sql
ALTER TABLE [ym].[Members] ADD [MiddleName] NVARCHAR(255) NULL;
```

CodeGen picks up the new column from `INFORMATION_SCHEMA` and creates the `EntityField` record.

### Field Type Change

```sql
ALTER TABLE [ym].[Members] ALTER COLUMN [Email] NVARCHAR(1000) NULL;  -- Increased length
```

CodeGen updates the `EntityField` record's type metadata.

### New Relationship

Schema Builder updates `additionalSchemaInfo` with the new soft FK entry. CodeGen applies it.

### Field Removal

Fields are **never physically dropped**. Instead:
1. The `CompanyIntegrationFieldMap` for that field is set to `Status = 'Inactive'`
2. The sync engine stops syncing the field
3. CodeGen marks the field description with `[DEPRECATED]` (via metadata JSON)
4. A future cleanup migration can drop the column after a user-defined grace period

Column drops in the generated migration are always **commented out** — they require manual uncommenting after review.

### Object Removal

Objects are never physically dropped. The `CompanyIntegrationEntityMap` is set to `Status = 'Inactive'`, and the sync engine stops syncing it.

## Table & Schema Naming

Target schema and table names are **user-controlled**, with smart defaults:

| Property | Default | Override |
|----------|---------|----------|
| **Schema** | Lowercase source type name (e.g., `hubspot`) | User can change to any valid schema name |
| **Table** | PascalCase singular of source object name (e.g., `Deal`) | User can rename |
| **Entity Name** | `{Schema} {TableLabel}` (e.g., `HubSpot Deal`) | User can rename |

Different source objects from the same integration can target **different schemas**. For example, HubSpot Contacts → `crm.Contact`, HubSpot Invoices → `billing.Invoice`.

## Safety & Pre-Flight Checks

Before generating artifacts, the Schema Builder validates:

1. **Schema collision check** — ensure schema name doesn't conflict with protected schemas
2. **Table collision check** — ensure table name doesn't collide with existing tables
3. **Entity name collision check** — ensure entity name doesn't conflict with existing MJ entities
4. **Blocklist check** — ensure target entity is not in the system entity blocklist
5. **Type compatibility** — ensure all source types can be mapped to the target platform

All schema operations are **forward-migration-only** — reversal is done by writing a new forward migration, never by rollback scripts. This ensures full traceability and reproducibility across environments.

## Soft Foreign Keys

### Why Soft FKs for Integrations

Integration data is notoriously unreliable for referential integrity:

- Source systems may not enforce FKs even when they "should"
- Sync timing means child records may arrive before parents
- Source IDs may reference records that were deleted in the source
- Cross-source relationships (HubSpot Deal → Salesforce Account) can't be guaranteed

MemberJunction's **soft FK system** solves all of these while still giving the framework full relationship awareness.

### How Soft FKs Work

Soft FKs are defined purely in metadata — no database constraint exists:

```
Database Layer:                    Metadata Layer (EntityField):
hubspot.Deal                       AccountID field:
  AccountID NVARCHAR(255) ───┐       RelatedEntityID = Account entity UUID
                             │       RelatedEntityFieldName = 'ID'
hubspot.Account              │       IsSoftForeignKey = 1
  ID UNIQUEIDENTIFIER  ◄────┘
       (no DB constraint)
```

**Framework behavior**: Soft FKs behave identically to hard FKs throughout MJ:
- GraphQL resolvers generate relationship fields
- Angular forms show dropdown lookups
- Record dependency detection works (for delete warnings)
- Record merging updates soft-FK-linked records
- Views include JOINs for denormalized field access

The only difference: the database doesn't enforce the constraint, so inserts/updates never fail due to FK violations.

### Configuration via `additionalSchemaInfo`

The Schema Builder updates the `additionalSchemaInfo` JSON file (path configured in `mj.config.cjs`):

```json
{
    "hubspot": [
        {
            "TableName": "Deal",
            "ForeignKeys": [
                {
                    "FieldName": "AccountID",
                    "SchemaName": "hubspot",
                    "RelatedTable": "Account",
                    "RelatedField": "ID"
                }
            ]
        },
        {
            "TableName": "Contact",
            "ForeignKeys": [
                {
                    "FieldName": "AccountID",
                    "SchemaName": "hubspot",
                    "RelatedTable": "Account",
                    "RelatedField": "ID"
                },
                {
                    "FieldName": "OwnerUserID",
                    "SchemaName": "__mj",
                    "RelatedTable": "User",
                    "RelatedField": "ID"
                }
            ]
        }
    ]
}
```

When CodeGen runs, it:
1. Reads `additionalSchemaInfo`
2. For each FK entry, sets `RelatedEntityID`, `RelatedEntityFieldName`, and `IsSoftForeignKey=1` on the corresponding `EntityField`
3. Emits the metadata changes to a CodeGen migration file
4. Validates for conflicts (e.g., if a hard FK already exists on the same field)

### Cross-Source Relationships

For relationships between integration schemas (e.g., HubSpot Deal → Salesforce Account), soft FKs are the only viable option. The Schema Builder allows users to define these, but sync timing may cause temporary broken references.

## Access Control

The `AccessControl` module enforces a three-tier security model for integration write targets:

```
User selects target entity
       │
       ▼
Entity in blocklist? ──Yes──→ BLOCKED (never allow)
       │ No
       ▼
Entity in __mj schema? ──No──→ ALLOWED (custom schema)
       │ Yes
       ▼
IntegrationWriteAllowed = true? ──Yes──→ ALLOWED
       │ No
       ▼
BLOCKED (show warning)
```

### Blocklist (Never Allow)

These entities are never writable by integrations, regardless of settings:

- `MJ: Entities`, `MJ: Entity Fields`, `MJ: Entity Relationships` — schema metadata
- `Users`, `Roles`, `User Roles` — security infrastructure
- `Applications`, `Application Entities` — app configuration
- `MJ: Entity Settings` — prevents recursive self-modification

The blocklist is enforced in code, independent of EntitySettings.

### Custom Schema Entities (Always Allowed)

Any entity in a non-`__mj` schema (e.g., `ym.Members`, `hs.Contacts`) is automatically allowed. These are integration-specific tables created by the Schema Builder.

### MJ Core Entities (Conditional)

Entities in the `__mj` schema require an explicit `IntegrationWriteAllowed=true` EntitySetting. Default allowlist (ships with MJ):

| Entity | Rationale |
|--------|-----------|
| `Lists` | Common target for marketing/CRM lists |
| `List Details` | Child records of Lists |
| `Contacts` | Core CRM data frequently synced from external sources |
| `Companies` | Core CRM data frequently synced from external sources |
| `Tags` | Categorization data from external systems |

## Migration File Naming

Files follow Flyway naming conventions:

```
V202603051200__v5.7_YourMembership_CreateTables.sql
V202603051201__v5.7_YourMembership_SoftFKConfig.sql
```

Format: `V{YYYYMMDDHHMM}__v{MJVersion}_{SourceType}_{Operation}.sql`

## Workflow: Adding Integration Tables

1. **Discover** — Use Connection Studio to discover objects and fields
2. **Configure** — In Mapping Workspace, select source objects and define target table names/schemas
3. **Preview** — View generated DDL in the UI (calls `IntegrationSchemaPreview`)
4. **Review** — Human reviews the migration SQL, soft FK config, and metadata
5. **Apply** — Run the migration: `flyway migrate`
6. **CodeGen** — Run CodeGen to generate entity classes: `npm run codegen`
7. **Map** — Configure field mappings in the Mapping Workspace
8. **Sync** — Run first sync to populate data
