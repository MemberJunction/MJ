Building Custom Apps

App-centric navigation, custom dashboards, lazy-loaded modules, and theming.

The shape of a custom app

An MJ application is a metadata record (Application) plus a set of resource components that render its custom screens. The metadata defines the app's name, icon, default nav items, and which entities it scopes. The components live in a feature module that's loaded on demand.

Most apps consist of:

  • A metadata file in /metadata/applications/
  • One or more BaseResourceComponent subclasses for custom screens (dashboards)
  • An Angular feature module that registers them
  • Optional: app-specific theme colors via the Application's color metadata

Custom dashboards

The bulk of custom-app work is dashboards. The Dashboard Best Practices guide walks through every architectural decision: state management with getter/setters, engine class patterns (no Angular services for data), user preferences, layout patterns, and permission checking.

Dashboard Best Practices →

Critical: every BaseResourceComponent subclass MUST call this.NotifyLoadComplete() when its initial load is finished. Without it, the app loading screen hangs forever on direct URL navigation. BaseDashboard handles this automatically; direct subclasses don't.

Lazy loading

MJExplorer uses code-split lazy loading for every dashboard, keeping the initial bundle small. Adding a new lazy-loaded component is zero-config: register the class with @RegisterClass, drop it in a feature module, and the auto-generated lazy config picks it up.

If you're authoring a new package that needs to be lazy-loadable, you'll add subpath exports to its package.json. The full pattern is in the guide.

Theming & design tokens

Every color in MJ comes from the semantic design token system in _tokens.scss. Never hardcode hex values — they break dark mode, prevent white-labeling, and create maintenance debt.

Per-application color customization flows from the database into CSS custom properties at runtime, so a single application can have its own brand without recompiling.

Custom entity forms

Generated entity forms cover most cases. When you need to override one, extend the generated form class (not BaseFormComponent directly) so the registration order resolves your custom form to higher priority.

Wrap your form's content in <mj-record-form-container> — not the bare toolbar. The container owns the panels that the History, Tags, and Add-to-List buttons open; without it, those features silently break.

App metadata files

To define an app, drop a JSON file in /metadata/applications/:

{
  "Name": "My App",
  "Description": "...",
  "DefaultForNewUser": false,
  "DefaultNavItems": [
    {
      "Label": "Home",
      "Icon": "fa-home",
      "ResourceType": "Custom",
      "DriverClass": "MyHomeResource",
      "isDefault": true
    }
  ]
}

Push with npx mj sync push. The DriverClass matches a class registered with @RegisterClass(BaseResourceComponent, 'MyHomeResource').
