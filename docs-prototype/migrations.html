Migrations

Versioned, repeatable, multi-platform schema evolution.

The model

MJ uses Flyway-style versioned migrations. Every schema change is a versioned .sql file that runs once, in order, on every environment. Files live in migrations/v5/ (the highest-numbered version folder is always the active one).

For PostgreSQL, MJ ships Skyway — a TypeScript implementation of the Flyway protocol — so the same versioned files run on either platform.

File format

VYYYYMMDDHHMM__v[VERSION].x_[DESCRIPTION].sql

# Example
V202604151430__v5.31.x_AddDashboardPermissions.sql

What CodeGen handles for you

Don't include any of the following in migration CREATE TABLE statements — CodeGen generates them automatically:

  • __mj_CreatedAt and __mj_UpdatedAt columns (and their triggers)
  • Indexes for foreign key columns (named IDX_AUTO_MJ_FKEY_*)
  • Stored procedures (spCreate*, spUpdate*, spDelete*)
  • Database views
  • Permission grants

You write business columns, constraints, and FKs. CodeGen does the rest. Manual entries here cause conflicts.

What you should always do

  • Use hardcoded UUIDs (not NEWID()) so the same migration produces identical IDs across environments
  • Use the ${flyway:defaultSchema} placeholder for schema-qualified references
  • Add an sp_addextendedproperty description for every new column (CodeGen uses these in generated docs)
  • Consolidate multiple ADD clauses into a single ALTER TABLE when adding multiple columns to the same table
  • Never insert __mj_CreatedAt / __mj_UpdatedAt values directly

Full conventions and examples: migrations/CLAUDE.md

Seed data: use metadata files, not SQL INSERTs

When a migration creates a new lookup or reference table, don't seed it with INSERT statements in the migration. Instead, create a metadata file in /metadata/ and let mj-sync push it.

This gives you:

  • Version-controlled, declarative, human-readable seed data
  • Automatic ID resolution via @lookup: references
  • Idempotent upsert semantics — safe to re-run
  • Consistency with how all other MJ reference data is managed

Example pattern: /metadata/resource-types/

The Publish-Then-No-Breaking-Changes Policy

Adopted 2026-04-29. Within a published OpenApp major version, only additive schema changes are allowed. No dropping tables/columns, no narrowing types, no renaming, no adding required parameters. Breaking changes force a major version bump.

Consult the full policy document before authoring any migration that modifies an existing schema.

Skyway: PostgreSQL migrations

Skyway is MJ's TypeScript implementation of Flyway-compatible database migrations, designed specifically for PostgreSQL workflows that need the same versioned-file model that the SQL Server side uses.
