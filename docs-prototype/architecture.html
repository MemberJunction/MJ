Architecture

Metadata-driven from the schema up. Pluggable providers from the AI layer down.

The core idea

Define your schema. MJ generates everything else.

Every layer of MJ — entity classes, GraphQL resolvers, validation, audit logging, Angular forms, even AI agent tool definitions — is derived from a single metadata model that lives in the database. CodeGen reads that metadata and produces the typed surface area developers work against.

This is what makes MJ different from a typical ORM or a typical UI framework: they're not separate systems wired together by hand. They're projections of the same metadata, kept in sync by code generation.

System layers

A live Mermaid diagram is rendered here on the production site. Source: MJ/README.md "Architecture" section.

Client layer. MJExplorer (Angular 21), MJCLI (command line), and MCP clients (Claude Desktop, Cursor, anything that speaks the Model Context Protocol).

API layer. A GraphQL server (@memberjunction/server) is the primary endpoint. An MCP server and an A2A (agent-to-agent) server expose the same data and capabilities to AI agents and other agentic platforms.

Core engine. The Metadata Engine (@memberjunction/core) is the heart. The Entity Framework gives you typed classes with validation, dirty tracking, and automatic audit. The Actions Engine handles workflows and scheduling.

AI layer. A provider-routing AI Engine, an Agent Framework with sub-agent delegation, vector ops (embeddings, semantic search, sync), and 15+ provider adapters behind a unified interface.

Data layer. SQL Server is primary; PostgreSQL via the GenericDatabaseProvider. Communication providers (email, SMS, push) live alongside.

Core concepts

Metadata-driven

Tables, fields, relationships, validation rules, value lists, and access control all live as metadata. Adding a column is a migration plus a CodeGen run — and the typed entity, resolver, form, and dropdown options all appear together.

Strong typing, no any

Generated entity classes give you full TypeScript IntelliSense for every field. The platform's contract is "you should never need as any" — if you do, the type system has a gap to fix, not bypass.

Provider pattern, everywhere

Data access, AI, communication, vectors, storage, auth — every external dependency is behind an interface with multiple implementations. Switch SQL Server to Postgres, OpenAI to Anthropic, SendGrid to Gmail, with config changes only.

BaseEntity event-driven invalidation

The server cache trusts itself completely because every entity mutation publishes events that downstream caches subscribe to. The result: RunView calls return cached data with zero DB round-trips when nothing relevant has changed. Caching & Pub/Sub guide →

Class registration without losing tree-shaking

Dynamic class instantiation (via @RegisterClass + MJGlobal.ClassFactory) is the platform's extension model. Modern bundlers can't see those references statically — so MJ ships a manifest system that emits a static CLASS_REGISTRATIONS array bundlers can't eliminate. Class manifest guide →

Audit, baked in

Record Changes is enabled by default for every entity. No bolted-on history tables, no custom triggers, no decisions to make per entity.

Design philosophy

What we lean into

  • Generation over abstraction
  • Strong types over runtime checks
  • Provider interfaces over hard-coded vendors
  • Event-driven invalidation over TTL guessing
  • Static manifests over dynamic discovery

What we avoid

  • any types and stringly-typed accessors
  • Re-exports between packages
  • Dynamic import() as a shortcut
  • Manual singletons (use BaseSingleton)
  • Hardcoded colors (use semantic design tokens)
