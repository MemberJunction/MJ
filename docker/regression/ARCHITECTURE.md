## 1. Overview

The MJ Agentic Test Runner is an **end-to-end browser-test harness** that
drives real applications with a headless Chromium instance controlled by an
LLM (the **Computer Use** engine). Tests are authored as MJ metadata JSON
files describing a natural-language *goal* (e.g. "log in and verify the
dashboard loads"). At runtime an LLM controller decides what to click, type,
or scroll based on screenshots of the current page; a second LLM judge
periodically evaluates whether the goal has been achieved; and deterministic
oracles (URL match, step count, custom plugins) produce a scored result.

Two callers consume this infrastructure:

- **The MemberJunction monorepo itself**, where it gates releases by
  running 25 canonical tests against a self-contained 5-container MJ stack
  every time `next` is built.
- **External adopters** вҖ” customers, internal teams, or any project вҖ” who
  point the published [`memberjunction/agentic-test-runner`][docker-hub-image]
  image at their own deployed app or bring up their app via a docker-compose
  overlay.

Both callers share the **same image** and **same testing framework**. The
difference is purely orchestration:

| Concern | MJ Monorepo | External Adopter |
|---|---|---|
| Source of test definitions | `metadata/tests/regression/*.json` | A scaffolded `metadata/tests/` dir generated by `init` |
| Stack being tested | Local docker MJ Explorer | A remote URL, or an app brought up by their own overlay |
| Results recording DB | Ephemeral docker SQL Server | The same or an external "runs DB" |
| Comparison baseline | Per-run `results.json` artifacts | Optionally an archive MJ instance |

The two key abstractions that enable the same code to serve both callers:

1. **Target profiles** (В§8) вҖ” JSON files describing *where to run* (URL,
   auth, suite name, archive destination). Decouple "what to test" from
   "where to test it."
2. **Variable substitution** (В§9) вҖ” Tests reference `{{baseUrl}}` /
   `{{authUsername}}` placeholders instead of hardcoded values. The same
   test JSON works across Mode A/B/C/D without modification.

[docker-hub-image]: https://hub.docker.com/r/memberjunction/agentic-test-runner

### Mode selector

The CLI infers the mode from the flags passed to `mj test regression`:

| Command | Mode |
|---|---|
| `mj test regression up` | A |
| `mj test regression remote --target=<remote-mj>` | B |
| `mj test regression remote --target=<generic-web>` | C |
| `mj test regression remote --target=<x> --overlay=<x>.yml` | D |

Under the hood, all four invoke `docker compose --profile <p> up` with
appropriate env vars and (for B/C/D) the remote entrypoint dispatcher.

## 4. Compose Profiles

Every service in [`docker-compose.test.yml`](docker-compose.test.yml) is
**profile-gated**. There is no "default" вҖ” running `docker compose up` with
no profile starts zero services (by design вҖ” a hard signal that the user
should go through `mj test regression *`).

| Profile | Services included | Mode |
|---|---|---|
| `full` | sqlserver, db-setup, mjapi, mjexplorer, test-runner | A; also B/C/D when the local stack records results |
| `remote-target` | test-runner only | B/C with `--no-local-stack` (external results DB) |
| `gen-forms` | sqlserver, form-generator | One-shot Angular form-codegen run |

`sqlserver` is in both `full` and `gen-forms` because both need a database
to bootstrap. `test-runner` is in both `full` and `remote-target` because
those are the only modes that run a suite.

### Why fully profile-gated

Before Phase 4 the services had no profiles вҖ” `docker compose up` brought
the whole stack online. That made it ambiguous which mode you were in if a
flag was missing. With everything profile-gated:

- The CLI is the single source of truth for "which profile to activate."
- Adding a new mode is one new profile string; no service-classification
  refactor required.
- A bare `docker compose up` failing fast (zero services) is a much
  better error than silently entering the wrong mode.

### `depends_on` semantics across profiles

`test-runner` has `depends_on: { mjapi, mjexplorer }` with
`required: false`. When the active profile is `remote-target` (where
`mjapi`/`mjexplorer` aren't in the profile), compose **skips the
dependency** instead of erroring. This is what lets the same service
definition serve both Mode A (waits for MJExplorer) and Modes B/C/D
(MJExplorer isn't running вҖ” proceed directly).

## 5. Container Inventory

### sqlserver

- **Image:** `mcr.microsoft.com/mssql/server:2022-latest` (platform: `linux/amd64`)
- **Host port:** 11433 вҶ’ container 1433
- **Profile:** `full`, `gen-forms`
- **Healthcheck:** `sqlcmd вҖҰ SELECT 1` every 10s
- **Volume:** *(none вҖ” DB is ephemeral; wiped on `down -v`)*
- **SA password:** from `TEST_SA_PASSWORD` env (default `TestPassword123!`)

### db-setup

- **Dockerfile:** [`Dockerfile.db-setup`](Dockerfile.db-setup) (FROM `Dockerfile.api`-shape)
- **Profile:** `full`
- **Lifecycle:** one-shot. Exits 0 on success, then mjapi can start.
- **Entrypoint:** [`db-setup-entrypoint.sh`](db-setup-entrypoint.sh)
- **Steps:**
  1. `bootstrap-db.cjs` вҖ” creates `MemberJunction_Test`, installs AssociationDB demo schema (58 tables, 10K+ rows)
  2. `mj migrate` вҖ” runs Flyway migrations to create `__mj` schema
  3. `mj codegen` вҖ” generates entity classes, views, stored procs, GraphQL resolvers, EntityField metadata
  4. `mj sync push` of `metadata/applications/`
  5. `mj sync push` of `metadata/prompts/` вҖ” picks up Computer Use controller/judge prompts
- **Bacpac mode** (`BACPAC_FILE` set, via `--bacpac` + the `docker-compose.bacpac.yml` overlay): the entrypoint branches to [`import-bacpac.cjs`](scripts/import-bacpac.cjs) (SqlPackage `/Action:Import` into `MemberJunction_Test`) вҶ’ optional `mj migrate` + `mj codegen` (gated by `BACPAC_UPGRADE`, default on; guarded against mis-baselining a DB with no `flyway_schema_history`) вҶ’ push prompts. AssociationDB + the standard app/test seeding are skipped; the user supplies their suite via `EXTRA_METADATA_DIRS`. The overlay pins db-setup to `linux/amd64` so the x64 SqlPackage build runs.
- **Why a separate container?** Migration + codegen + AssociationDB install takes ~2-3 minutes. Keeping them out of MJAPI's Dockerfile keeps the runtime image lean and lets `service_completed_successfully` block dependents until the DB is fully ready.

### mjapi

- **Dockerfile:** [`Dockerfile.api`](Dockerfile.api)
- **Host port:** 14000 вҶ’ container 4000
- **Profile:** `full`
- **Depends on:** `db-setup` (with `condition: service_completed_successfully`)
- **Entrypoint:** runs Express/Apollo via `npm start`
- **Restart policy:** `on-failure:5` вҖ” auto-restart on OOM under heavy suite load
- **Build-time customization:** [`scripts/patch-test-api-config.cjs`](scripts/patch-test-api-config.cjs) flips `autoCreateNewUsers`, `newUserRoles`, and `CreateUserApplicationRecords` for the test environment
- **Healthcheck:** `curl /healthcheck` every 10s

### mjexplorer

- **Dockerfile:** [`Dockerfile.explorer`](Dockerfile.explorer) (multi-stage)
- **Host port:** 4200 вҶ’ container 4200
- **Profile:** `full`
- **Build args:** `AUTH0_DOMAIN`, `AUTH0_CLIENTID` (Angular AOT build bakes them in)
- **Runtime:** nginx:alpine serving the built Angular app + reverse-proxying `/api/` to `http://mjapi:4000/`
- **nginx config:** [`nginx.conf`](nginx.conf) вҖ” uses a `map $http_upgrade $connection_upgrade` block to conditionally set the `Connection` header so GraphQL POSTs don't hang on a phantom WebSocket upgrade
- **Build-time customization:** [`scripts/gen-environment-ts.cjs`](scripts/gen-environment-ts.cjs) generates `src/environments/environment.ts` from build args

### test-runner

- **Dockerfile:** [`Dockerfile.test-runner`](Dockerfile.test-runner)
- **Base:** `mcr.microsoft.com/playwright:v1.58.1-noble` (Playwright + Chromium)
- **Profile:** `full`, `remote-target`
- **Depends on:** `mjapi` (healthy), `mjexplorer` (started) вҖ” both `required: false` so the runner can run alone in `remote-target` profile
- **Entrypoint:** **selected by env var** at compose-parse time вҖ” `entrypoint: ["/bin/bash", "/app/${TEST_RUNNER_ENTRYPOINT:-test-runner-entrypoint.sh}"]`
- **Volumes** (bind-mounted from the host):
  - `./test-results:/app/test-results` (RW вҖ” output dir)
  - `./test-runner-entrypoint.sh:/app/test-runner-entrypoint.sh:ro`
  - `./test-runner-remote-entrypoint.sh:/app/test-runner-remote-entrypoint.sh:ro`
  - `./scripts:/app/docker/regression/scripts:ro` вҖ” `.cjs` helpers
  - `./archive:/app/docker/regression/archive:ro` вҖ” archive entity-cascade configs
  - `./targets:/app/docker/regression/targets:ro` вҖ” target profiles
  - `./examples:/app/docker/regression/examples` (RW вҖ” mj-sync writes back primaryKey/sync)
  - `../../metadata:/app/metadata` (RW)
  - `./test-metadata:/app/test-metadata:ro` вҖ” Docker-only test user/tags
- **Shared memory:** `shm_size: 4gb` (Chromium tab stability)
- **Bootstrap:** `NODE_OPTIONS="--import /app/bootstrap.mjs"` ensures `@RegisterClass` fires before the CLI runs

### form-generator (opt-in)

- **Profile:** `gen-forms` only
- **Reuses:** `Dockerfile.db-setup` (same monorepo build, same `mj` CLI)
- **Purpose:** runs codegen against a temporary stack to produce Angular entity-form components into `.docker-generated/MJExplorer-forms/` on the host
- **Why separate?** Angular forms are generated from DB schema, and the DB must exist for codegen to run. That setup can't run inside a Docker `build` step. So the form-generator spawns a temp stack, runs codegen, captures output to a host bind mount, and tears itself down. The output is then COPYed into `mjexplorer` at image-build time.
- **Lifecycle:** one-shot. Runs only when explicitly invoked via `mj test regression gen-forms` (or the build-time guard in `mj test regression build`).

### byo-app (Mode D overlay-supplied)

- **Defined by:** the user's overlay (e.g. [`examples/bring-your-own-app/docker-compose.app.yml`](examples/bring-your-own-app/docker-compose.app.yml))
- **Profile:** `full` (joins the `--profile full` startup)
- **Added by overlay** to the existing compose project; not present in the base
- **Conventional service name:** anything вҖ” the runner reaches it via the name set in the target profile's `baseUrl`

## 6. Entrypoint Dispatch

The `test-runner` service uses an **env-var-driven entrypoint selector** in
the compose YAML:

```yaml
test-runner:
  entrypoint: ["/bin/bash", "/app/${TEST_RUNNER_ENTRYPOINT:-test-runner-entrypoint.sh}"]
```

This lets the same container image run two completely different orchestration
flows without rebuilding:

| `TEST_RUNNER_ENTRYPOINT` value | Script | When used |
|---|---|---|
| *(unset вҖ” default)* | `test-runner-entrypoint.sh` | Mode A (self-contained MJ stack) |
| `test-runner-remote-entrypoint.sh` | `test-runner-remote-entrypoint.sh` | Modes B/C; sometimes D |

The CLI sets `TEST_RUNNER_ENTRYPOINT` from the target profile's loader output
(see В§8). The Mode D path inherits the env var setting if the target profile
loaded set it; otherwise the overlay's hardcoded env vars may override.

### Full-stack entrypoint вҖ” `test-runner-entrypoint.sh`

Used for Mode A. Big steps:

1. **socat shim:** `socat TCP-LISTEN:4200,fork,reuseaddr TCP:mjexplorer:4200 &` вҖ” Auth0's `auth0-spa-js` requires a secure origin, and browsers treat `localhost` as secure but **not** arbitrary Docker hostnames like `mjexplorer`. socat forwards traffic so the browser thinks it's hitting `localhost:4200`.
2. **Application metadata sync:** `mj sync push --dir=metadata --include="applications"`
3. **Test-user metadata sync:** `mj sync push --dir=/app/test-metadata --include="tags,users"`
4. **SQL safety-net:** `scripts/setup-test-user.cjs` direct SQL upserts for the test user, roles, applications, and example data вҖ” guards against `mj sync` failing on first run (chicken-and-egg)
5. **Test/suite metadata sync:** `mj sync push --dir=metadata --include="tests"` then `вҖҰinclude="test-suites"`
6. **Extra metadata dirs (optional):** for Mode D / BYO вҖ” push anything in `EXTRA_METADATA_DIRS` (comma-separated)
7. **Preflight diagnostics:** `scripts/preflight-checks.cjs` probes MJAPI/nginx/socat/Auth0 + memory snapshot вҶ’ `preflight.json`
8. **Per-run output dir:** `test-results/run-{TIMESTAMP}/` with `screenshots/`
9. **Background health monitor:** `scripts/health-monitor.cjs` writes `diagnostics.json` every 10s while the suite runs
10. **Run the suite:** `npx mj test suite --name "${TEST_SUITE_NAME:-MJ Explorer Regression Suite}" --parallel --max-parallel ${MAX_PARALLEL_WORKERS:-4}` (with `--oracles-module=вҖҰ` if `ORACLES_MODULE` is set)
11. **Stop health monitor**
12. **Extract screenshots** from `__mj.vwTestRunOutputs` вҶ’ `screenshots/<test>/step_NN.png`
13. **Generate markdown report:** `scripts/generate-md-report.cjs`
14. **Generate HTML report:** `scripts/generate-html-report.cjs` (lightbox gallery)
15. **Optional archive flow:** if `ARCHIVE_DB_DATABASE` is set, pull the suite run + children to JSON, tag, push to destination MJ
16. **Update `test-results/latest` symlink**

### Remote entrypoint вҖ” `test-runner-remote-entrypoint.sh`

Used for Modes B/C. Mirror of the above with these differences:

| Skipped | Why |
|---|---|
| socat shim | Target is reachable directly at `MJ_TEST_VAR_baseUrl` |
| Application metadata sync | No MJ Explorer to seed |
| Test-user metadata sync | No local DB to provision |
| `setup-test-user.cjs` | No local DB |

## 8. Target Profile System

A **target profile** is a JSON file describing *where to run* the suite. It
keeps environment-specific config (URLs, credentials, archive destinations)
out of test definitions, so the same test JSON works against multiple
deployments without modification.

### Schema

```jsonc
{
  "schemaVersion": "1.0",                            // required, currently always "1.0"
  "name": "staging-mj",                              // required, short identifier (used for bare-name lookup)
  "kind": "mj-explorer",                             // required: mj-explorer | generic-web | mj-api-only
  "description": "...",                              // optional, free-form

  // в”Җв”Җв”Җ What URL to test в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
  "baseUrl": "https://staging-mj.example.com",       // required
  "allowedDomains": ["staging-mj.example.com",       // optional, array of host patterns
                     "*.auth0.com"],

  // в”Җв”Җв”Җ Auth (optional вҖ” empty bindings for unauthed apps) в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
  "auth": {
    "username": "env:STAGING_TEST_USER",             // resolved from host env
    "password": "env:STAGING_TEST_PASSWORD",         // resolved from host env
    "domains": ["staging-mj.example.com",
                "*.auth0.com"]
  },

  // в”Җв”Җв”Җ Which suite to run в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
  "suite": "MJ Explorer Regression Suite",           // optional; defaults to MJ suite

  // в”Җв”Җв”Җ Extra metadata directories to push before the suite runs в”Җв”Җ
  "extraMetadataDirs": [                             // optional
    "/app/my-app-metadata"
  ],

  // в”Җв”Җв”Җ Custom oracles в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
```

### Loader вҖ” `scripts/load-target-profile.cjs`

Runs **host-side** (invoked by the CLI). Steps:

1. Read the JSON file at the given path.
2. Validate required fields (`name`, `kind`, `baseUrl`).
3. Validate `kind` is one of: `mj-explorer`, `generic-web`, `mj-api-only`.
4. For each `env:VARNAME` reference, look it up in `process.env`. Exit 1 if missing.
5. Auto-JSON-stringify array-typed values (`allowedDomains`, `auth.domains`) before exporting them.
6. Emit env-var assignments in one of two formats:
   - `--format=env` (default): shell `export FOO='bar'` lines, POSIX-safe (single-quote escape for embedded quotes).
   - `--format=json`: a single JSON object with all key/value pairs.

The CLI uses `--format=json` (cleaner parsing in TypeScript); a developer
debugging can use `eval "$(node load-target-profile.cjs my.json)"` for
inspection.

### Env-var translation

| Target field | Emitted env var |
|---|---|
| `baseUrl` | `MJ_TEST_VAR_baseUrl` |
| `allowedDomains` | `MJ_TEST_VAR_allowedDomains` (JSON-stringified array) |
| `auth.username` | `MJ_TEST_VAR_authUsername` (env-ref resolved) |
| `auth.password` | `MJ_TEST_VAR_authPassword` (env-ref resolved) |
| `auth.domains` | `MJ_TEST_VAR_authDomains` (JSON-stringified array) |
| `suite` | `TEST_SUITE_NAME` |
| `extraMetadataDirs` | `EXTRA_METADATA_DIRS` (comma-joined) |
| `oraclesModule` | `ORACLES_MODULE` |
| `archive.tag` | `ARCHIVE_TAG` |
| `archive.source` | `ARCHIVE_SOURCE` |
| *(always)* | `TEST_RUNNER_MODE=remote`, `TEST_RUNNER_ENTRYPOINT=test-runner-remote-entrypoint.sh`, `TARGET_PROFILE_NAME`, `TARGET_PROFILE_KIND` |

### Resolution by short name vs. path

`mj test regression remote --target=<value>`:

- If `<value>` contains a `/` or ends with `.target.json` вҶ’ treated as a path
- Otherwise вҶ’ resolved to `docker/regression/targets/<value>.target.json`

This is what lets `mj test regression remote --target=byo-app` work as a
shorthand for the full file path.

## 9. Variable Substitution

Test JSONs reference `{{key}}` placeholders that get resolved at runtime
against a values map. Lives in
[`packages/AI/MJComputerUse/src/utils/variable-substitution.ts`](../../packages/AI/MJComputerUse/src/utils/variable-substitution.ts).

### Two sources, two precedences

```
buildVariableValuesFromContext(context, env=process.env) вҶ’ values: Record<string, unknown>

  Layer 1 (low precedence):  env vars matching MJ_TEST_VAR_*
                             вҖў key = env name minus "MJ_TEST_VAR_" prefix
                             вҖў JSON-parse the value if it looks structured

  Layer 2 (overrides):       context.resolvedVariables.values
                             вҖў set by `--var key=value` CLI flags
                             вҖў or by the TestingFramework's VariableResolver
                               (with schema validation)
```

Layer 2 wins on conflict. Most callers (the docker entrypoint, BYO overlay,
target-profile loader) populate Layer 1 via env vars; the resolver is for
developers passing `--var` flags directly.

### Substitution semantics

`substituteVariables(obj, values)` is recursive and pure (returns a new object).

- **Whole-string placeholder** (`"{{baseUrl}}"`) вҖ” replaced with the **raw value** from the map, preserving non-string types. E.g. `{{allowedDomains}}` resolves to an array, not a string.
- **Embedded placeholder** (`"see {{baseUrl}}/path"`) вҖ” always produces a string; non-string values are coerced via `String()`.
- **Unknown key** вҖ” left in place verbatim. No error. This lets authors ship partially-parameterized tests.

### Auto-JSON-parsing for env-var Layer 1

A real-world quirk: env vars are always strings. If `MJ_TEST_VAR_allowedDomains` is `'["a","b"]'`, the substitution would inject the **literal string** `'["a","b"]'` into an array-typed field. The loader detects "looks like JSON" (starts with `[`, `{`, `"`, or a digit / boolean / null) and `JSON.parse()`s it before storing. Falls back to the raw string on parse failure or plain text.

### Where substitution fires

The driver applies substitution at three points before calling the engine:

```typescript
// packages/AI/MJComputerUse/src/test-driver/ComputerUseTestDriver.ts
const values = buildVariableValuesFromContext(context);
const config = substituteVariables(test.Configuration, values);
const input = substituteVariables(test.InputDefinition, values);
const expected = substituteVariables(test.ExpectedOutcomes, values);
```

Plus the suite-level applicationContext (В§11) and any oracle `pattern` configs.

## 10. Custom Oracles

An **oracle** is a deterministic check that evaluates the test's output and
produces a `{ passed, score, message, details? }` result. Each test
configures one or more oracles (with weights); the weighted average becomes
the test's final score.

### Built-in oracles (Computer Use)

| Type | Source | What it checks |
|---|---|---|
| `goal-completion` | [`GoalCompletionOracle.ts`](../../packages/AI/MJComputerUse/src/test-driver/oracles/GoalCompletionOracle.ts) | The judge LLM verdict `Done=true` + confidence вүҘ minConfidence |
| `url-match` | [`UrlMatchOracle.ts`](../../packages/AI/MJComputerUse/src/test-driver/oracles/UrlMatchOracle.ts) | The final URL matches a regex pattern |
| `step-count` | [`StepCountOracle.ts`](../../packages/AI/MJComputerUse/src/test-driver/oracles/StepCountOracle.ts) | The agent finished within maxSteps |

Built-in oracles for non-Computer-Use test types live in
[`packages/TestingFramework/Engine/src/oracles/`](../../packages/TestingFramework/Engine/src/oracles/)
and cover schema validation, LLM-judge, exact-match, SQL validation, and trace validation.

### The IOracle interface

```typescript
export interface IOracle {
    readonly type: string;                         // unique id (e.g. 'final-url-host')
    evaluate(input: OracleInput,                   // { actualOutput, expectedOutput, targetEntity, contextUser }
             config: OracleConfig): Promise<OracleResult>;
}
```

`OracleResult` shape:

```typescript
{
    oracleType: string;     // mirrors this.type
    passed: boolean;
    score: number;          // 0.0вҖ“1.0
    message: string;        // human-readable
    details?: unknown;      // optional structured context for reports
}
```

### Plugging in custom oracles вҖ” `--oracles-module`

External adopters write their own `IOracle` implementations and load them at
suite startup:

```bash
mj test suite --oracles-module=path/to/my-oracles.cjs --name "My Suite"
```

Or via a target profile's `oraclesModule` field (works through
`mj test regression remote`).

The loader at
[`packages/TestingFramework/CLI/src/utils/oracle-module-loader.ts`](../../packages/TestingFramework/CLI/src/utils/oracle-module-loader.ts)
**duck-types** each export вҖ” anything with `type: string` and an async
`evaluate()` gets registered. Helpers, constants, unrelated exports are
silently skipped.

Two export styles are both supported:

```javascript
// Class style вҖ” instantiated with no args
class FinalUrlHostOracle {
    constructor() { this.type = 'final-url-host'; }
    async evaluate(input, config) { /* вҖҰ */ }
}

// Instance style вҖ” used as-is
const myOracle = {
    type: 'my-oracle',
    async evaluate(input, config) { /* вҖҰ */ },
};

module.exports = { FinalUrlHostOracle, myOracle };
```

Reference implementation at
[`docker/regression/examples/generic-web/oracles/mdn-oracles.cjs`](examples/generic-web/oracles/mdn-oracles.cjs).

### Registration order + override

Custom oracles registered via the loader land in the **engine-level**
registry. Computer Use's three built-ins are scoped to the driver's
`builtInOracles` map (which **takes precedence** over engine-level oracles
of the same type). So a custom `goal-completion` plugin would NOT override
the driver's; a custom `final-url-host` would be unique and would register.
This boundary is by design вҖ” built-ins are part of the test contract.

## 11. Suite-Level Application Context

A suite can declare a markdown blob of "what the controller LLM should know
before the first step" via `MJTestSuiteEntity.Configuration.applicationContext`.
This appears in the controller prompt on every step (Anthropic prompt-caching
amortizes the cost across the run), helping the agent skip rediscovery of
the app's navigation landmarks, common gotchas, etc.

### Wiring

1. **Suite metadata:** `metadata/test-suites/.regression-suite.json` has a
   `Configuration` field (JSON string column) with shape:
   ```jsonc
   {
     "applicationContext": "## MJ Explorer\n\nThe top nav has...\n### Common gotchas\n- Wait up to 30s for transitions...\n"
   }
   ```
2. **TestEngine.RunSuite** parses `suite.Configuration` JSON, extracts
   `applicationContext`, attaches it to `DriverExecutionContext.suiteContext`.
3. **ComputerUseTestDriver.Execute** reads `context.suiteContext.applicationContext`,
   applies `{{var}}` substitution (so it can reference `{{baseUrl}}`),
   concatenates with the per-test override (`test.InputDefinition.applicationContext`)
   under a `## Test-specific Notes` heading.
4. **ComputerUseEngine** forwards `params.ApplicationContext` into the
   controller prompt request on every step.
5. **Controller prompt template**
   ([`metadata/prompts/templates/computer-use/controller.template.md`](../../metadata/prompts/templates/computer-use/controller.template.md))
   has a `{% if applicationContext %}` block that renders the context under
   a `## Application Context` heading.

### Composition rules

`composeApplicationContext(suiteLevel, perTest, values)` in
[`variable-substitution.ts`](../../packages/AI/MJComputerUse/src/utils/variable-substitution.ts):

- Returns `undefined` when both layers are empty (engine then skips the
  heading entirely вҖ” no empty section rendered).
- Returns just the suite layer when only that exists.
- Returns just the per-test layer (under `## Test-specific Notes`) when only
  that exists.
- Returns both joined when both exist, suite first.
- Applies `{{var}}` substitution to each layer before joining.

> **Current state:** the plumbing is fully wired and exercised by the BYO
> suite (`Configuration.applicationContext` is 853 chars). The canonical
> MJ Explorer Regression Suite **does not yet have an applicationContext
> authored** вҖ” see [LIMITATIONS.md В§2](LIMITATIONS.md#2-suite-level-application-context-phase-1b).

## 12. Browser Context Isolation

Each test runs in a **fresh Playwright `BrowserContext`** seeded with the
worker's cached `storageState` (cookies + localStorage). This avoids:

- **Test pollution** вҖ” cookies set in test 5 don't leak into test 6.
- **Cross-test side effects** вҖ” IndexedDB / sessionStorage entries from one
  test don't trip oracles in another.
- **Heuristic state-cleanup bugs** вҖ” the old shared-context cleanup pass
  used regex matching on localStorage keys (e.g. `/auth0|msal|okta/`) and
  was fragile; non-matching custom auth schemes got nuked.

### How it stays fast (auth replay)

A naive "fresh context per test" approach would re-trigger Auth0's login
flow (~10-20s) on every test. To avoid this:

1. After the **first** test in a worker successfully authenticates,
   `HeadlessBrowserEngine` calls `context.storageState()` and caches the
   result keyed by worker index.
2. On **subsequent** tests in the same worker, the new context is created
   with `browser.newContext({ storageState: cached })` so cookies + localStorage
   are pre-populated.
3. The `AuthHandler` sees the user is already logged in and short-circuits вҖ”
   no Auth0 round-trip needed.

### Token expiry handling

If a cached `storageState` becomes stale mid-suite (Auth0 token expired,
which happens on hour-long suites with default token TTLs), the page lands
on the login screen despite injection. The driver detects this and calls
`engine.InvalidateStorageState(workerKey)`, letting `AuthHandler` re-run
once; the new state is then captured and used for the rest of the worker's
tests.

### Opt-in shared sessions

The default `browserSession` strategy is `"new"` (isolated per test). Two
opt-ins remain for rare tests that genuinely need cross-test state:

- `browserSession: "shared:suite"` вҖ” single context across the worker's
  whole suite. Use when test N+1 verifies an artifact created by test N.
- `browserSession: "shared:global"` вҖ” single context across all workers.
  Almost never the right choice; degrades parallelism.

Both opt-ins emit a one-time warning when used.

## 13. Archive Flow

Optional pipeline that forwards a completed suite run + all child records to
a destination MJ instance (an "archive MJ"). Triggered when the
`ARCHIVE_DB_DATABASE` env var is set. Uses `mj-sync` for the wire protocol вҖ”
no custom sink code.

### Configuration contract

The archive flow is **env-var driven** вҖ” no files to copy, no `.cjs` to edit.
A transient `mj.config.cjs` is generated at runtime from the vars below.

| Env var | Required? | Default | Notes |
|---|---|---|---|
| `ARCHIVE_DB_DATABASE` | yes | вҖ” | destination database name |
| `ARCHIVE_DB_USERNAME` | yes | вҖ” | destination SQL user |
| `ARCHIVE_DB_PASSWORD` | yes | вҖ” | destination SQL password |
| `ARCHIVE_DB_HOST` | no | `host.docker.internal` | works for most setups; override for in-network or remote destinations |
| `ARCHIVE_DB_PORT` | no | `1433` | |
| `ARCHIVE_DB_TRUST_CERT` | no | `true` | |
| `ARCHIVE_USER_EMAIL` | no | `not.set@nowhere.com` | user that owns the archived records; must exist in destination |
| `ARCHIVE_MJ_CORE_SCHEMA` | no | `__mj` | schema name on destination |
| `ARCHIVE_TAG` | no | `<TEST_SUITE_NAME>` | written into `TestSuiteRun.Tags` |
| `ARCHIVE_SOURCE` | no | empty | written into `TestSuiteRun.MachineName` |

### Network topology cheat-sheet

| Where is the destination? | What to set |
|---|---|
| Another docker container on its own network | `docker network connect mj-regression_default <ctr>` then `ARCHIVE_DB_HOST=<ctr>` |
| Your host machine (Mac/Windows/Linux) | keep `host.docker.internal` default + ensure destination publishes its port to the host |
| A remote server | `ARCHIVE_DB_HOST=<fqdn-or-ip>` + open the firewall |

The test-runner compose service declares `extra_hosts: host.docker.internal:host-gateway`,
so the default works portably on Linux too (where Docker Desktop's auto-mapping doesn't apply).

### Archive entity-cascade config

[`docker/regression/archive/test-suite-runs/.mj-sync.json`](archive/test-suite-runs/.mj-sync.json)
defines what to pull and how to externalize:

- **3-level cascade:** `MJ: Test Suite Runs` вҶ’ `MJ: Test Runs` вҶ’ `MJ: Test Run Outputs`
- **`excludeFields`** at each level strips joined virtual columns (`Suite`, `Test`, `RunByUser`, `OutputType`) that aren't writable on push and would cause validation errors.
- **`externalizeFields`** on `TestRunOutput.InlineData` writes base64 PNGs as files (`screenshots/{TestRunID}/{ID}.png`) instead of inlining them in the JSON. Keeps JSON files small and human-readable; survives a JSON round-trip.

| File | Use |

| [`scripts/archive-run.sh`](scripts/archive-run.sh) | The post-suite archive orchestrator (config-gen вҶ’ pre-seed вҶ’ pull вҶ’ tag вҶ’ push). **Sourced** by both `test-runner-entrypoint.sh` (В§8) and `test-runner-remote-entrypoint.sh` (В§5) so the logic lives in one place |
| [`scripts/archive-preflight.cjs`](scripts/archive-preflight.cjs) | Pre-flight validator. Runs ~5s before the suite; fails fast on misconfig |
| [`scripts/archive-config-gen.cjs`](scripts/archive-config-gen.cjs) | Generates `/tmp/mj-archive-config.cjs` from `ARCHIVE_DB_*` env vars |
| [`scripts/tag-archive.cjs`](scripts/tag-archive.cjs) | Writes `Tags` + `MachineName` into the pulled archive JSON before push |

The `MJ_CONFIG_FILE` env override (added in
[`packages/MetadataSync/src/lib/config-manager.ts`](../../packages/MetadataSync/src/lib/config-manager.ts))
lets `mj sync push` use a non-default config without changing cwd вҖ” the
pull uses the local config (cosmiconfig defaults), the push uses the
explicit one.

**вҡ пёҸ Common pitfall:** never set `MJ_CONFIG_FILE` as a container-wide env var
(e.g., via `docker run -e MJ_CONFIG_FILE=...`). It must be scoped to the push
command only, because the pull MUST use the local docker DB config. The
entrypoint applies it inline for that reason; a container-wide setting would
redirect the pull and silently produce an empty archive.

### Tag-based filtering on the destination

The destination MJ accumulates runs from multiple sources (staging-nightly,
customer-x-monthly, etc.). `mj test compare --tag=staging-nightly` filters
the DB-mode comparison query to only runs whose `Tags` array contains the
tag. See В§14.

## 17. Diagnostic Outputs

Every run creates `docker/regression/test-results/run-{TIMESTAMP}/` with:

| File | Contents |
|---|---|
| `results.json` | Machine-readable suite results вҖ” emitted by `mj test suite --format=json`. Top-level keys: `suiteRunId`, `suiteName`, `status`, `passedTests`, `failedTests`, `totalTests`, `averageScore`, `testResults[]`, `durationMs`, `totalCost`, `startedAt`, `completedAt` |
| `report.md` | Human-readable markdown summary вҖ” generated by `scripts/generate-md-report.cjs` |
| `report.html` | Self-contained HTML gallery with lightbox screenshots + action overlays + oracle results table вҖ” generated by `scripts/generate-html-report.cjs` |
| `preflight.json` | Pre-run diagnostics: MJAPI health, GraphQL reachability, nginx proxy, socat TCP, Auth0 OIDC discovery, memory snapshot. Captures the environment state at run start. |
| `diagnostics.json` | Background health monitor log вҖ” probes every 10s during the suite |
| `screenshots/<test-name>/` | Per-step PNGs + `steps.json` (reasoning + actions + URL per step) |
| `archive/` | Only when `ARCHIVE_DB_DATABASE` is set вҖ” the pulled entity-cascade JSON tree + externalized PNGs ready for `mj sync push` |

Plus a `latest` symlink at `docker/regression/test-results/latest вҶ’
run-{TIMESTAMP}/` for convenience.

### Comparing runs

`mj test regression compare` auto-discovers the two newest `run-*/results.json`
files by mtime and produces a diff. Default output is a console table; with
`--format=json` or `--format=markdown` you get machine-parsable or
report-friendly variants.

Exit codes are CI-friendly:

| Code | Meaning |
|---|---|
| 0 | No regressions detected |
| 1 | One or more regressions detected (use `--diff-only` to see them) |
| 2 | Data error (fewer than 2 runs found, missing files, invalid JSON) |

Definition of "regression": previous status was `Passed` and current is not,
OR score dropped by more than 0.10. "Improvement" is the inverse.

## 18. Extension Points

### Add a new regression test

1. Create `metadata/tests/regression/.T26-<short-name>.json` mirroring the shape of an existing test (e.g. T01).
2. Add a `relatedEntities[].MJ: Test Suite Tests` entry to `metadata/test-suites/.regression-suite.json` linking the new test to the suite.
3. `mj test regression up` will pick up both via the bind-mounted `metadata/` volume вҖ” no rebuild needed.

### Add a new oracle type

1. Implement `IOracle` (TS or JS file). See `mdn-oracles.cjs` for a template.
2. Ship it in your repo (or copy into a project-specific scaffold).
3. Reference it in the test JSON's `Configuration.oracles[]` array.
4. Load it via `--oracles-module=path/to/file.cjs` (or `oraclesModule` in a target profile).

### Add a new target kind

1. Add the kind name to `SUPPORTED_KINDS` in `scripts/load-target-profile.cjs`.
2. (Optional) Add kind-specific validation logic in the loader.
3. (Optional) Branch entrypoint behavior on `TARGET_PROFILE_KIND` if needed (e.g. an `mj-api-only` kind that skips browser launch entirely).

### Add a new compose profile

1. Add `profiles: ["<name>"]` to relevant services in `docker-compose.test.yml`.
2. Add a new CLI subcommand in `packages/MJCLI/src/commands/test/regression/` if the profile needs a user-facing entry point.
3. Document in `REGRESSION_TESTING.md` and (if new mode) in В§2 of this file.

### Add a new example scaffold

1. Create `docker/regression/examples/<name>/` with at minimum a `target.json` and a `README.md`.
2. (If Mode D) include `docker-compose.app.yml` + the app's Dockerfile.
3. (If using custom oracles) include `oracles/<name>-oracles.cjs`.
4. (If shipping tests) include `metadata/{tests,test-suites}/` mirroring the canonical layout.
5. Add a row to `docker/regression/examples/README.md`.
6. Add a row to В§15 of this file.
7. The next image build will bake it into `/app/examples/`, and `init` will be able to extract it.

### Make a package lazy-loadable in the runner image

The published image bundles `dist/` of every `@memberjunction/*` package. Trim the image size by:

1. Marking your package's expensive dependencies as `optionalDependencies` or `peerDependenciesMeta`.
2. Using dynamic `import()` for runtime-discovered modules (the existing `mj test regression` CLI does this for testing-cli).

See the [Dynamic Import policy in CLAUDE.md](../../CLAUDE.md#8-no-dynamic-import-unless-narrowly-justified) for the rules вҖ” runtime plugin discovery is one of the allowed exemptions.

## 19. Glossary

| Term | Definition |
|---|---|
| **Agentic test** | A test whose execution is decided at runtime by an LLM controller, not a hardcoded script. The Computer Use engine is the agentic part. |
| **Computer Use** | Anthropic's tool-use mode where the LLM produces screenshot-anchored actions (click coordinates, key presses, text inputs). MJ wraps this in [`packages/AI/ComputerUse/`](../../packages/AI/ComputerUse/). |
| **Controller** | The LLM that decides the next step (one prompt per step). Anchored by the goal + screenshot history. |
| **Judge** | The LLM that periodically evaluates whether the goal is achieved. Configurable cadence via `Configuration.judgeFrequency`. |
| **Oracle** | A deterministic post-run evaluator. Goal-completion, url-match, step-count are built-in; plugins via `--oracles-module`. |
| **Driver** | A `BaseTestDriver` subclass that executes one test type. `ComputerUseTestDriver` is the one used by the regression suite. |
| **Target profile** | JSON describing where to run (URL, auth, suite, archive). See В§8. |
| **Mode A/B/C/D** | The four run modes. See В§2. |
| **AssociationDB** | The demo dataset baked into the docker DB. 58 tables, 10K+ rows of members/events/courses/certifications/etc. Provides realistic UI to test against. |
| **Suite run** | One execution of `mj test suite`. Produces a `MJTestSuiteRun` row + N `MJTestRun` rows + M `MJTestRunOutput` rows. |
| **Test output** | A persisted artifact from one step вҖ” typically a screenshot stored as base64 in `TestRunOutput.InlineData`. |
| **Variable substitution** | The `{{key}}` placeholder system. See В§9. |
| **Archive flow** | The optional `mj sync pull вҶ’ tag вҶ’ push` pipeline that forwards a suite run to a destination MJ. See В§13. |
| **Bare name** | A target profile reference without a path (e.g. `byo-app`), resolved to `targets/<bare>.target.json`. |
| **Overlay** | A compose YAML file that extends the base via `-f base -f overlay`. Used for Mode D to add the byo-app service + test-runner deps. |
| **storageState** | Playwright's snapshot of cookies + localStorage for a `BrowserContext`. Captured + replayed per worker to avoid re-authenticating between tests. |
| **Externalization** (mj-sync) | Writing a large field (`InlineData`) to disk as a separate file (`screenshots/<id>.png`) and replacing the in-JSON value with `@file:` reference. Keeps JSONs small + diffable. |
