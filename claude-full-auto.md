# Claude Full Autonomy Development Guide

> **WARNING: SANDBOXED / AIR-GAPPED ENVIRONMENTS ONLY**
>
> This approach is designed EXCLUSIVELY for sandboxed, air-gapped, or Docker-based development machines where Claude has full system access. **DO NOT** use this approach on production machines, shared development environments, or any machine with access to production data or services.
>
> The standard co-pilot approach (see [CLAUDE.md](CLAUDE.md)) should be used for all regular development workflows. This guide extends that with full-autonomy patterns for isolated environments.

---

## What Is Full Autonomy Development?

In full autonomy mode, Claude operates as an independent developer on a sandboxed machine with:
- Full database access (create/drop databases, run migrations, run CodeGen)
- Full build pipeline access (npm install, npm run build, compile, test)
- Full browser testing access (Playwright E2E, headed/headless)
- No human approval gates except for explicitly defined checkpoints

This differs from the standard co-pilot model where Claude assists a human developer who owns the build/deploy cycle. In full autonomy, Claude owns the entire cycle end-to-end.

---

## Environment Requirements

- **Isolated SQL Server**: Local Docker container or dedicated dev instance. Never a shared or production database.
- **Sandboxed machine**: No access to production APIs, services, or data. Network access limited to package registries (npm) and documentation.
- **Dedicated branch**: Always work on a feature branch. Never push directly to `next` or `main`.
- **Git as the safety net**: All work is committed incrementally. If anything goes wrong, `git log` and `git diff` tell the full story.

## Standard Full-Autonomy Workflow

### 1. Environment Setup
- Create a new database for the feature branch (naming: `MJ_<branch_short_name>`)
- Update `.env` with the new database name
- Ensure the branch is up to date with `next`
- Run `mj migrate` to bring the new DB to current state

### 2. Migration Development
- Write the feature migration SQL in `migrations/v5/` using the standard naming format: `VYYYYMMDDHHMM__v[VERSION].x_[DESCRIPTION].sql`
- **Checkpoint**: Present migration to human for review before executing
- After approval: run `mj migrate` to apply the migration

### 3. CodeGen Cycle
- Run `mj codegen` after migration
- CodeGen emits SQL output files (typically `CodeGen_Run_*.sql` in `migrations/v5/`)
- Append the CodeGen output to the migration file:
  1. Add ~50 blank lines after the migration SQL
  2. Add a comment block clearly marking the CodeGen output boundary
  3. Use unix commands (`cat`, append redirection) to combine — no inference for this mechanical step
  4. Run `mj migrate` again to apply the CodeGen-emitted SQL

### 4. Implementation
- Build packages, write TypeScript, create Angular/React components
- Run `npm run build` in affected packages to verify compilation
- Run unit tests: `cd packages/PackageName && npm run test`
- Fix any compilation or test failures immediately

### 5. E2E Testing
- Start MJAPI and MJExplorer as background processes
- Use Playwright CLI with persistent auth profile for browser testing
- Test the full user-facing flow end-to-end
- Capture and fix any issues found during testing

### 6. Commit and Push
- Stage specific files (never `git add -A` in autonomy mode — be explicit)
- Commit with descriptive messages
- Push to the feature branch for PR review

## Database Management

### Creating a New Database
```bash
# Create via sqlcmd in Docker
docker exec -it <sql-container> /opt/mssql-tools18/bin/sqlcmd \
  -S localhost -U sa -P '<password>' -C \
  -Q "CREATE DATABASE [MJ_geo_features]"

# Or via npm/node if sqlcmd isn't available
# Use the mssql npm package to connect and execute
```

### Naming Convention
- Format: `MJ_<feature_short_name>` (e.g., `MJ_geo_features`)
- Keep it short but descriptive
- One database per feature branch when possible

### Cleanup
- Drop the database when the feature branch is merged
- Document any persistent test data that might be useful

## Migration + CodeGen Append Pattern

After CodeGen runs, the output SQL must be appended to your migration file. This is a mechanical operation — use shell commands, not inference:

```bash
# Add whitespace and comment block separator
printf '\n%.0s' {1..50} >> migrations/v5/YOUR_MIGRATION.sql
cat >> migrations/v5/YOUR_MIGRATION.sql << 'COMMENT'
-- ================================================================
-- CODEGEN OUTPUT
-- Auto-generated by MJ CodeGen. Do not manually edit below this line.
-- Appended on: <date>
-- ================================================================
COMMENT

# Append the CodeGen SQL output
cat migrations/v5/CodeGen_Run_YYYY-MM-DD_HH-MM-SS.sql >> migrations/v5/YOUR_MIGRATION.sql
```

## Checkpoints (When to Stop and Ask)

Even in full autonomy mode, certain decisions require human input:
1. **Migration SQL review** — before first execution against the database
2. **Architecture decisions** that deviate from the approved plan
3. **Destructive operations** on shared resources (even in sandbox, confirm before dropping databases with data)
4. **Scope changes** — if implementation reveals the plan needs significant revision
