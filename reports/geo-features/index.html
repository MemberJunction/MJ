Executive Summary

This report documents the first fully autonomous development session on a sandboxed Mac, implementing the Geo Features architecture plan for MemberJunction. The session covered all four phases of the plan: database infrastructure, server-side packages, Angular and React UI components, scheduled geocoding, and reference data seeding.

All work was done on branch an-bc/geocoding against a dedicated MJ_geo_features SQL Server database created specifically for this session.

Architecture Overview

WRITE PATH (infrequent) Record Save (address changed) → Generated subclass fires GeoCodeSyncService.SyncIfChanged() (fire-and-forget) → Computes hash, checks RecordGeoCode for staleness → If stale: upserts row with Status='pending', dispatches geocoding → On success: Status='success' with lat/lng, CountryID, StateProvinceID → On failure: Status='failed' with ErrorMessage, increments RetryCount → Save ALWAYS succeeds regardless of geocoding outcome READ PATH (fast, no API calls) Map View / Grid / Smart Filter → Standard RunView call → Base view includes __mj_Latitude, __mj_Longitude via LEFT JOIN → Map component renders markers/choropleth/heatmap → Zero geocoding API calls at read time GAP FILLER (scheduled) ScheduledGeocodingAction (daily/weekly) → Finds pending, failed, and missing RecordGeoCode rows → Batch geocodes with global rate limiter (50 req/sec) → Fills all gaps from bulk imports, bypassed Save paths
