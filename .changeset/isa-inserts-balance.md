Fix `generateCRUDCreate` emitting a duplicate primary-key column in the generated `spCreate` `INSERT` statement when the entity has a single UUID PK with no database-level default — most commonly an IS-A (table-per-type) child entity whose `ID` is both PK and FK to the parent, where the parent (or `BaseEntity.NewRecord()`) mints the UUID rather than the child table carrying its own default. Direct follow-on from `curious-pkeys-emerge`: that change correctly added an `isCallerSuppliedPK` exception so PK columns appear in the INSERT for composite and string-PK entities, but it also caused the PK to appear twice for the single-UUID-no-default branch — once via the field iteration and once via the `additionalFieldList` injection this branch had been doing all along, producing `Msg 264: The column name 'ID' is specified more than once...` and making the generated `spCreate` invalid SQL. The fix passes `excludePrimaryKey=true` to the two `generateInsertFieldString` calls in the fallback INSERT block (matching the pattern of the `hasDefaultValue` branch above and the composite-PK branch below), so the iteration suppresses the PK and `additionalFieldList`/`additionalValueList` provide it exactly once. Two regression tests added asserting `[ID]` appears exactly once in the `INSERT` column list and `@ActualID` (not raw `@ID`) is the only PK value in the `VALUES` list. Validated end-to-end against a real Azure SQL Database with an IS-A child of `__mj.ContentItem`.
