## 2025-04-18 - Stack Trace Visibility in Error Boundaries
**Learning:** Hiding stack traces in UI error boundaries (while leaving them in the console) is not a real security improvement and actively degrades the user experience by making it harder to copy/paste errors for debugging.
**Prevention:** Do not blindly hide stack traces in internal or administrative UIs. Security improvements should focus on actual vulnerabilities (like XSS or SQLi), not security theater that hinders usability.

## 2025-04-18 - XSS Vulnerability in UI Highlighting functions
**Vulnerability:** The `HighlightMatch` functions in `search-results.component.ts` and `dashboard-browser.component.ts` were taking raw string input and manipulating them with regular expressions to output HTML strings with `<mark>` tags, which were then bound to `[innerHTML]`. This created a Cross-Site Scripting (XSS) vulnerability.
**Learning:** This existed because the custom highlighting function only considered the use case of wrapping matching text with an HTML tag for styling purposes and overlooked that the resulting string would be used directly in an insecure `innerHTML` binding. Without sanitization or escaping of the original input string, any HTML elements including malicious script tags were evaluated by the browser. Additionally, a naive approach of escaping first and regex matching second breaks HTML entities.
**Prevention:** Whenever rendering HTML programmatically and using `[innerHTML]`, always explicitly escape HTML special characters (`<`, `>`, `&`, `"`, `'`) or use a DOM sanitizer library on user inputs. When building strings that mix escaped content and unescaped HTML tags (like `<mark>`), split the original unescaped text first, escape the individual segments, and then assemble the final string to avoid double-escaping or breaking HTML entities.
